# Screens: 06 Supplier Onboarding and Qualification (Build-real)

Flow 06 realized as screens. Source: `documentation/06-supplier-onboarding.md`, data dictionary `Supplier.json` / `SupplierAddress.json` / `SupplierAddressTaxDetail.json` / `SupplierGroup.json` / `SupplierScorecard.json`, `model/role-permission-matrix.md`, state machine `supplierLifecycle` (PENDING_ONBOARDING -> PENDING_APPROVAL -> ONBOARDED -> OFFBOARDED; plus ONBOARDED <-> SUSPENDED). Built to active supplier (with profile + scorecard).

### S06.1 Suppliers (archetype: list)
- **Route:** `/suppliers`
- **Roles:** Procurement/Buyer (act: create, update, import); Quality (act: AVL grade, suspend on quality grounds); Tax/Compliance (act: tax/ISO attributes, suspend on compliance grounds); Approver (act: approve, offboard); Management (view); Administrator (view all + act).
- **Realizes:** entry point; the supplier master view; the "AVL grade spread + status badges" overview.
- **Purpose:** the supplier base at a glance: who is orderable, who is pending, who is SUSPENDED (blocked from new POs), and the AVL grade distribution, plus the entry to add one supplier or bulk-import many.
- **Data shown:** `Supplier.code`, `Supplier.name`, `Supplier.classification` (Internal | External), `Supplier.groupId`, `Supplier.currency`, `Supplier.status` (Badge per `supplierLifecycle`, including SUSPENDED), `Supplier.avlGrade` (Badge A/B/C/D/F), AVL status (Approved | Conditional | Preferred | Suspended | Disqualified), `Supplier.riskTier`, latest `SupplierScorecard.composite` / `grade` where available, certification-expiry warning where a cert is near/past expiry. A KPI strip shows the AVL grade spread and count by status.
- **Primary actions:** "New Supplier" (-> S06.2; perm `suppliers.create`); "Bulk Import" (-> S06.4; perm `suppliers.import`); row click -> supplier detail (S06.3).
- **Secondary actions / navigation:** faceted filter by status / AVL grade / classification / group / risk tier; search by code / name; sort; pagination (default 10, max 50); link to supplier scorecard (12-analytics).
- **States to design:** empty ("No suppliers yet"), loading (skeleton + KPI tiles), populated, error, permission-denied.
- **Shadcn components:** DataTable, Card (AVL grade-spread / status-count KPI tiles), Badge (status incl. SUSPENDED, AVL grade, risk tier, cert-expiry), Popover + Command (faceted filter), Input (search), Button.
- **Acceptance criteria (believable done):** (1) `Supplier.status` Badges use the canonical `supplierLifecycle` states and a SUSPENDED supplier is visibly distinguished (it blocks new POs); (2) the AVL grade-spread KPI reflects real `Supplier.avlGrade` values across the base, not a static chart; (3) a supplier with a certification near or past `expiryDate` shows an expiry warning Badge; (4) "New Supplier" and "Bulk Import" are gated by `suppliers.create` / `suppliers.import`; (5) empty state shows for a fresh persona.
- **Tags:** [S1.6 | ISO 8.4.1 | AB AVL + RA lifecycle]

### S06.2 Create / Edit Supplier (archetype: form / wizard)
- **Route:** `/suppliers/new`, `/suppliers/:id/edit`
- **Roles:** Procurement/Buyer (act: `suppliers.create`, `suppliers.update`, request approval); Quality (act: AVL scope/grade, `supplier.qualify` quality criteria); Tax/Compliance (act: tax registration + ISO attributes, `supplier.riskAssess`).
- **Realizes:** narrative steps 3 (create profile), 4 (capture tax + registration), 5 (upload qualification docs + certs), 6 (set AVL scope/grade), 7 (set ISO attributes), 8 (request approval).
- **Purpose:** capture the full qualification package that takes a supplier from PENDING_ONBOARDING toward ONBOARDED: profile, addresses with tax registration, qualification documents and certifications with expiry, AVL scope and grade, and the cross-standard ISO attribute set.
- **Data shown / captured (grouped sections / wizard steps):**
  - PROFILE from `Supplier.json` - `Supplier.code` (auto S/#####, read-only), `Supplier.name`, `Supplier.currency` (Select), `Supplier.dealCurrency` (Select, defaults to currency), `Supplier.classification` (Select, Internal | External), `Supplier.groupId` (Combobox over `SupplierGroup.name`), `Supplier.paymentTerms` (Select, PaymentTerms enum), `Supplier.minimumOrderValue`, `Supplier.advancePayable` (toggle), `Supplier.advanceTolerance`, `Supplier.autoInvoicing` (toggle), `Supplier.status` (Badge, read-only).
  - ADDRESSES from `SupplierAddress.json` (>= 1, replace-on-update) - `addressCode` (default A1), `line1` / `line2` / `line3`, `city`, `state`, `zip`, `country`, `gstin` (regex-validated India GST when supplied), `pan` (regex-validated India PAN when supplied); per-address tax detail from `SupplierAddressTaxDetail.json` - `taxRegistration`, `stateCode` (padded to 2 digits, must agree with the GSTIN leading digits), `taxRegion`.
  - QUALIFICATION DOCS - `Supplier.vendorRegistration` (file), `Supplier.threeBatchCOA` (files, mandatory for OTC/pharma RM/PM), `Supplier.certifications` (repeatable object: certType, certNumber, issuingBody, issueDate, expiryDate mandatory, evidence file), MSDS / permits (files, for chemicals / regulated chemicals).
  - AVL - `Supplier.avlScopeOfApproval` (text, categories/items approved to supply), `Supplier.avlGrade` (Select A/B/C/D/F with AVL status Approved | Conditional | Suspended | Disqualified | Preferred), approval date, next re-eval date.
  - ISO ATTRIBUTE SET - `Supplier.riskTier` (Select, RiskTier; ISO 31000), `Supplier.securityTradeProgram` (Select, TradeProgram; ISO 28000), `Supplier.continuity` (ISO 22301 object: criticalityFlag, bcPlan, MTPD, RTO, MBCO, RPO, backupSupplierRegister, dualSourcingCoverage), `Supplier.sustainabilityESG` (ISO 20400 object: EcoVadis/CDP, ghgData, modernSlaveryStatement, codeOfConductSignoff, conflictMinerals), `Supplier.iso14001` (object: status, scope), `Supplier.infoSecDPA` (ISO 27001 object: dataClassificationAccessed, securityCertStatus, dpaStatus, cloudAttestations), `Supplier.antiBriberyDDscreening` (ISO 37001 object: briberyRiskRating, dueDiligenceStatus, beneficialOwnership/PEP/sanctions screening, antiBriberyClauseSignoff, reScreeningCadence); each ISO group carries last-assessed / next-due / evidence + expiry / CAPA log.
- **Primary actions:** "Save Draft" (creates/updates at PENDING_ONBOARDING; no gate); "Request Approval" (`supplierLifecycle` PENDING_ONBOARDING -> PENDING_APPROVAL; perm via Buyer; guard status = PENDING_ONBOARDING; fires the SUPPLIER_APPROVAL_REQUESTED notification only here). On an ONBOARDED/PENDING_APPROVAL record, saving an edit reverts status to PENDING_APPROVAL and resets `Supplier.isErpSynced` (axiom A10).
- **Secondary actions / navigation:** add/remove address; add/remove certification; add/remove ISO evidence; the cert-expiry inline alert (see inline behaviours); link to supplier detail (S06.3).
- **States to design:** new (empty, status PENDING_ONBOARDING), edit (fetch, status reverts to PENDING_APPROVAL on save), validation-error (malformed GST/PAN, missing mandatory cert expiry, missing >= 1 address, missing 3-batch COA for an OTC/pharma supplier), saved-draft toast, approval-requested toast, permission-denied.
- **Shadcn components:** Stepper (Profile -> Addresses and Tax -> Qualification and Certs -> AVL -> ISO attributes) or Form with Card sections + Tabs, Select/Combobox (currency, group, classification, payment terms, AVL grade, risk tier, trade program), Sheet (address add/edit, certification add/edit), file Input, DatePicker (cert expiry, re-eval), Toggle (advancePayable / autoInvoicing), Toast.
- **Acceptance criteria (believable done):** (1) `Supplier.code` is generated once (S/##### pattern) and shown read-only, immutable across edits; (2) `gstin` and `pan` are validated by regex only when supplied and a malformed value blocks save with a field error; the `SupplierAddressTaxDetail.stateCode` is padded to two digits and must agree with the GSTIN leading digits; (3) every certification requires an `expiryDate`; (4) "Request Approval" requires status = PENDING_ONBOARDING and moves to PENDING_APPROVAL, firing the SUPPLIER_APPROVAL_REQUESTED notification only here (not at approve/offboard); (5) editing an ONBOARDED or PENDING_APPROVAL supplier reverts the status to PENDING_APPROVAL and resets `Supplier.isErpSynced` to false (axiom A10), and the address/tax-detail set is replaced wholesale (not merged); (6) the 3-batch COA section is required for an OTC/pharma RM/PM supplier and certifications act as a hard pass/fail gate (axiom A15), not a weighted score.
- **Tags:** [S1.6 | ISO 8.4.1 / 7.5 + 31000/28000/22301/20400/27001/37001 | AB profile + RA lifecycle]

### S06.3 Supplier Detail (archetype: detail)
- **Route:** `/suppliers/:id`
- **Roles:** Procurement/Buyer (act: edit, request approval, offboard); Approver (act: approve, offboard); Quality (act: suspend on quality grounds, AVL grade); Tax/Compliance (act: suspend on compliance grounds); Management (view scorecard); Administrator (act: suspend always, all lifecycle).
- **Realizes:** narrative steps 9-12 (review qualification, approve, mark onboarded, optional ERP sync), 13-18 (post-onboarding lifecycle: edit, suspend, restore, offboard), and the scorecard view.
- **Purpose:** the single record of truth for one supplier across its lifecycle: profile, addresses and tax, qualification and certifications, AVL, ISO attributes, the performance scorecard, and the lifecycle actions, with only the legal transitions shown.
- **Data shown:** all profile + address + tax + qualification + AVL + ISO fields; `Supplier.status` (Badge) and the `supplierLifecycle` position; `Supplier.isErpSynced`; certification expiry status; the latest `SupplierScorecard` (composite, the four sub-scores deliveryScore / qualityScore / costScore / responsivenessScore with stored weights, grade, AVL status, complianceGate pass/fail, two-factor OTIF, four-factor perfect order) and the period trend; the append-only audit trail; linked POs / RFQs / NCRs / CAPAs.
- **Primary actions (state-aware, per `supplierLifecycle`):** "Request Approval" (Buyer; PENDING_ONBOARDING -> PENDING_APPROVAL); "Approve" (Approver, `suppliers.approve`; PENDING_APPROVAL -> ONBOARDED; guard: qualification complete - certs/COA/AVL, ISO attributes for risk tier, sanctions clear); "Suspend" (Quality / Tax/Compliance / Admin, `suppliers.suspend`; ONBOARDED -> SUSPENDED; guard: failed audit / expired cert / sanctions hit / active CAPA; mandatory suspendReason + suspendCategory: quality | compliance | expired-cert | sanctions | active-CAPA; effect: blocks new POs); "Reinstate" (Quality / Tax/Compliance / Admin; SUSPENDED -> ONBOARDED; guard: issue resolved); "Offboard" (Approver / Admin, `suppliers.offboard`; ONBOARDED -> OFFBOARDED, terminal); "Edit" (-> S06.2; reverts to PENDING_APPROVAL); "Toggle ERP Sync" (only when ONBOARDED). Buttons render ONLY for legal transitions.
- **Secondary actions / navigation:** Tabs (Profile / Addresses and Tax / Qualification and Certs / AVL and ISO / Scorecard / Audit); link to the scorecard dashboard (12-analytics), POs (05), RFQs (04), NCR/CAPA (11).
- **States to design:** loading, pending-onboarding (Request Approval shown), pending-approval (Approve/Return shown for Approver), onboarded (Suspend/Offboard/Edit/ERP-toggle shown), suspended (Reinstate shown; new-PO blocked banner), offboarded (terminal, read-only), permission-denied.
- **Shadcn components:** Card + Tabs, Badge (status incl. SUSPENDED, AVL grade/status, compliance gate, cert-expiry), Dialog (suspend with reason + category, offboard confirm, reinstate), Toggle (ERP sync), charts (Recharts for scorecard sub-scores and trend), Button group, Toast.
- **Acceptance criteria (believable done):** (1) only legal `supplierLifecycle` transition buttons render for the current status (no Approve unless PENDING_APPROVAL; no Suspend unless ONBOARDED; no Offboard on a SUSPENDED record); (2) Approve requires qualification complete (certs valid/unexpired, 3-batch COA for OTC/pharma, AVL scope/grade set, sanctions clear, required ISO attributes for the risk tier); (3) Suspend is split by grounds - Quality on a quality stop, Tax/Compliance on a sanctions/compliance hit, Admin always - and a SUSPENDED supplier shows a "blocks new POs" banner (axiom A11) and cannot be awarded an RFQ or issued a PO; (4) the scorecard shows the composite as the weighted roll-up of the four sub-scores with compliance as a separate pass/fail gate (axiom A15, never folded into the composite), and the two-factor OTIF and four-factor perfect order are labelled distinctly (axiom A14); (5) editing reverts to PENDING_APPROVAL and resets `isErpSynced`; the ERP-sync toggle is available only when ONBOARDED; (6) the audit trail (category SUPPLIER) is append-only and timestamped.
- **Tags:** [S1.6 / OE3 | ISO 8.4.1 re-evaluation | AB lifecycle + review SUSPENDED + scorecard]

### S06.4 Bulk Import Suppliers (archetype: modal / form)
- **Route:** `/suppliers/import`
- **Roles:** Procurement/Buyer (act: `suppliers.import`); Administrator.
- **Realizes:** the bulk branch (upload import file, validate header + rows, all-or-nothing upsert by code).
- **Purpose:** load many suppliers from one XLSX/CSV with strict all-or-nothing validation, so a partial/dirty import never half-writes the master.
- **Data shown / captured:** file Input (XLSX/CSV); a downloadable template with the exact header; a pre-commit validation result table listing every `{rowNumber, reason}` on any error; on success, a summary of inserted/updated counts (upsert by natural key = `Supplier.code`, duplicates last-wins, default status PENDING_ONBOARDING).
- **Primary actions:** "Upload and Validate" (validates header + rows, accumulates all errors); "Commit Import" (enabled only when zero errors; one transaction; audit emitted only on commit).
- **Secondary actions / navigation:** download template; cancel; after commit, link to the supplier list (S06.1) where imported rows appear at PENDING_ONBOARDING needing docs + attributes + request-approval.
- **States to design:** empty (no file), validating (loading), errors (every row error listed, commit disabled), valid (commit enabled), committed (summary + redirect), error.
- **Shadcn components:** Dialog or Sheet, file Input, DataTable (per-row validation results), Alert (error summary), Button, Toast.
- **Acceptance criteria (believable done):** (1) the import is all-or-nothing - any file/header/row error rejects the entire import and writes nothing, listing every `{rowNumber, reason}` (e06); (2) the header must match exactly; a header-only file reports "No valid data found", no file reports "No file uploaded"; (3) upsert is by natural key = `Supplier.code` with duplicates last-wins, default status PENDING_ONBOARDING; (4) audit is emitted only on a successful commit (a late DB failure rolls back fully with no audit); (5) imported suppliers still flow through docs + ISO attributes + request-approval -> approve to reach ONBOARDED.
- **Tags:** [e06 | RA bulk import all-or-nothing]

### Inline edge behaviours shown on these screens (not separate screens)
- **Certification expiry alert:** on S06.1 / S06.2 / S06.3 a certification at or within one week of `expiryDate` raises an expiry alert; an expired cert is a suspend ground (ONBOARDED -> SUSPENDED).
- **Edit reverts to re-approval (axiom A10):** on S06.2 / S06.3 saving any edit on an ONBOARDED/PENDING_APPROVAL supplier silently recomputes status to PENDING_APPROVAL and resets `isErpSynced`; the address and tax-detail sets are replaced wholesale.
- **Suspend blocks new POs (axiom A11):** on S06.3 a SUSPENDED supplier shows a blocking banner and is excluded from RFQ award (04) and PO issue (05) until reinstated.
- **Compliance is a hard gate, not a score (axiom A15):** on the scorecard view, cert validity + document completeness gate AVL eligibility separately from the weighted composite.
- **CAPA-to-suspend loop:** an active CAPA (from 11/e04) feeds the scorecard and can drive ONBOARDED -> SUSPENDED with suspendCategory = active-CAPA.
