# Screens: 01 Configuration and Master Data (Show-light, Administrator)

Flow 01 realized as screens. Show-light depth: prove the tenant is configurable (masters, RBAC, approval chains/routing rules, dynamic field config), do not build the full setup wizard. Four representative screens cover the configuration surface; bulk import (e06) surfaces as an upload action on the master-data screen with all-or-nothing result. Source: `documentation/01-configuration.md`, `mvp-and-storyboard.md` section 2 (Show-light), `model/role-permission-matrix.md`, data dictionary `Currency.json` / `UoM.json` / `Warehouse.json` / `PaymentTerms.json` / `Project.json` / `Category.json` / `Item.json` / `Supplier.json` / `TaxCode.json` / `User.json` / `Role.json` / `Permission.json` / `Designation.json` / `BusinessUnit.json` / `RoutingRule.json` / `FieldConfig.json` / `Budget.json`. State machines: master creates set Supplier/Item `status==PENDING_ONBOARDING` (supplierLifecycle / itemLifecycle initial); routing rules and field config feed the approvalCompletion and requisitionStage machines.

All four screens are gated to the Administrator (super-admin `all`, or the seeded SystemAdmin role). Every committed change emits an audit entry and an SSE event; rejected imports emit neither (A13).

### S01.1 Master Data Management (archetype: list / workbench with per-master tabs)
- **Route:** `/admin/masters`, `/admin/masters/:master` (master one of currencies, units, warehouses, payment-terms, projects, categories, items, suppliers, tax-codes).
- **Roles:** Administrator (act: `masters.manage`); Tax/Compliance owns tax-code fields in the model but the screen is Admin-gated for the prototype.
- **Realizes:** configuration steps 3-9: create masters individually (4a) or by bulk import (4b), validation + code generation (5), the all-or-nothing bulk gate (6), reject-with-row-errors (7), correct-and-re-upload (8), upsert in one transaction (9).
- **Purpose:** prove every master is maintainable (create, edit, bulk import) from one screen so the tenant's reference data is demonstrably configurable, not hard-coded.
- **Data shown:** one tab per master, each a DataTable of that master's records.
  - **Currency:** `Currency.isoCode`, `name`, `symbol`, `isBase` (Badge), `decimals`, `status`.
  - **UoM:** `UoM.uom`, `fullForm`, `category`, `status`.
  - **Warehouse:** `Warehouse.code`, `name`, `location`, `type`, `status`.
  - **PaymentTerms:** `PaymentTerms.code`, `label`, `termType`, `advancePercent`, `netDays`, `triggerEvent`, `status`.
  - **Project/CostCenter:** `Project.code`, `name`, `type`, `status`, `budgetId`, `managerId`.
  - **Category/Segment:** `Category.classType`, `classCode`, `classDesc`, `segment`, `subSegment`, `requisitionCategory`, `status` (composite natural key classType + classCode + classDesc).
  - **Item:** `Item.code`, `description`, `stockUom`, `purchaseUom`, `categoryId`, `hsCode`, `status` (lifecycle Badge, default PENDING_ONBOARDING).
  - **Supplier:** `Supplier.code`, `name`, `currency`, `classification`, `groupId`, `paymentTerms`, `status` (lifecycle Badge, default PENDING_ONBOARDING), `avlGrade`.
  - **TaxCode:** `TaxCode.code`, `type`, `rate`, `jurisdiction`, `recoverable`, `isReverseCharge`, `status`.
- **Primary actions:**
  - "Create" (per master; opens a Sheet form with that master's fields and validation; perm `masters.manage`). On save, per-record validation runs, the code is generated (Item `code`, Supplier `code` S/#####), references resolve, and Supplier/Item are persisted as PENDING_ONBOARDING (supplierLifecycle / itemLifecycle initial state). No transition on the other masters (they have no lifecycle). A bad record returns a 400 and writes nothing for that record.
  - "Edit" (per row; opens the same Sheet pre-filled; perm `masters.manage`). Natural keys (`isoCode`, `uom`, codes, the Category composite) are read-only on edit (immutable). Editing a Supplier/Item that is ONBOARDED moves it back to PENDING_APPROVAL and sets `isErpSynced=false` (A10); shown only for those two masters.
  - "Bulk import" (per master; opens an upload Dialog; perm `masters.manage`, surfaces e06). Upload one CSV/XLSX for the selected master into the generic import pipeline. The header is validated exactly (no missing/extra columns, order-insensitive); rows are validated and references resolved by natural key; all row errors accumulate. All-or-nothing: if `rowErrors.length == 0` the file upserts in one transaction (match-by-natural-key updates, else create with generated code; duplicate keys in one file resolve last-wins; imported Supplier/Item default PENDING_ONBOARDING; `isErpSynced` resets on update). If `rowErrors.length > 0` the entire import is rejected with a row-numbered reason list and nothing is written.
- **Secondary actions / navigation:** search and faceted filter per master; sort; pagination; "Set as base" on Currency (sets `isBase`; exactly one base per tenant; locked once any transaction exists); deactivate/reactivate (`status` ACTIVE/INACTIVE) where the master has a status.
- **States to design:** empty (per master, "No records. Create the first or import."), loading (skeleton table), populated, validation-error (inline per field on the create/edit Sheet), import-result (success toast with upserted count, OR a reject panel listing every {rowNumber, reason}), permission-denied.
- **Shadcn:** Tabs (one per master), DataTable, Sheet (create/edit form), Form (react-hook-form + Zod per master), Select/Combobox (references: currency, UoM, project, category), Dialog (bulk-import upload + result), file Input, Badge (status, isBase, PENDING_ONBOARDING), Button, Toast, Alert (import reject list).
- **Acceptance criteria:** (1) bulk import is all-or-nothing: a file with any invalid row writes nothing and shows the full {rowNumber, reason} list; a clean file upserts every row in one transaction (e06 / A-bulk-import); (2) a bulk header with a missing or extra column rejects the whole file before any row is processed; (3) created/imported Supplier and Item records land in PENDING_ONBOARDING (lifecycle initial), shown by the status Badge, never auto-ONBOARDED; (4) natural keys (isoCode, uom, codes, the Category composite) are read-only on edit (immutable); (5) exactly one Currency carries `isBase==true`, and base cannot be changed once a transaction exists; (6) editing an ONBOARDED Supplier/Item flips `isErpSynced=false` and routes it to PENDING_APPROVAL (A10).
- **Tags:** [OE4 | ISO 7.5 | data-model masters + raphe-bulk-import]

### S01.2 Roles, Permissions and User Assignment (archetype: detail / matrix view + assignment)
- **Route:** `/admin/rbac`, with tabs Roles, Permission Matrix, Users.
- **Roles:** Administrator (act: `roles.manage`, `users.create`, `users.view`).
- **Realizes:** configuration steps 10-12: create users/designations/business units, define roles and grant permissions (RBAC), and the effective-permission resolution model (union of four sources + super-admin short-circuit).
- **Purpose:** prove RBAC is real and editable: roles carry namespaced permissions, users resolve to an effective set, and the matrix is the same one the nav and every action gate read from.
- **Data shown:**
  - **Roles tab:** `Role.name`, `description`, `isSystem` (Badge; SystemAdmin/User are system roles, cannot be deleted), permission count. SystemAdmin shows the single `all` super-permission.
  - **Permission Matrix tab:** a read/edit grid of `Permission.value` (grouped by `Permission.resource`) x roles, mirroring `model/role-permission-matrix.md`. Each cell is granted/not for that role's `Role.permissions`. `Permission.description` shows on hover; `Permission.isAdminOnly` permissions (notably `all`) are visually flagged and grantable only by a holder of `all`.
  - **Users tab:** `User.name`, `email`, `roleId`, `designationId` (rank), `departmentId`, `businessUnitId`, `status` (ACTIVE/INACTIVE), `approvalLimit`, direct `userPermissions` count. A computed "effective permissions" panel per user shows the deduplicated union of role + direct grant + designation + business-unit, plus the `all` short-circuit.
- **Primary actions:**
  - "Create role" / "Edit role" (Sheet; set `name`, `description`, `permissions` via a permission picker grouped by resource; perm `roles.manage`). System roles (`isSystem==true`) cannot be deleted.
  - "Toggle permission" in the matrix (grants/revokes a `Permission.value` on a `Role`; perm `roles.manage`; admin-only permissions require the actor to hold `all`).
  - "Create user" / "Edit user" (Sheet; set `email`, `name`, `departmentId`, `designationId`, `businessUnitId`, `roleId`, `approvalLimit`, `userPermissions`, `status`; perm `users.create`). `email` is trimmed, lowercased, unique case-insensitive. `approvalLimit` null reads as the default 200000; a configured 0 stays 0.
  - "Activate / Deactivate user" (sets `User.status`; INACTIVE forces logout on the next request and removes the user from assignee/approver eligibility; perm `users.create`).
  - "Set SoD ruleset" (toggle the SoD rules: buyer != PO approver, receiver != invoice approver, maker != checker, requester != sole approver of own requisition; on by default; relaxing any rule is recorded in the audit log; perm `roles.manage`).
- **Secondary actions / navigation:** search users; filter by role/department/status; group the matrix by `Permission.resource`; view the audit entry for a permission change.
- **States to design:** loading, populated, validation-error (duplicate email, blank required field), permission-denied (a non-`all` holder attempting to grant an admin-only permission is blocked with a reason), empty (no custom roles yet; shows the seeded SystemAdmin and User).
- **Shadcn:** Tabs, DataTable (roles, users), bespoke matrix Table (permissions x roles, checkbox cells), Sheet + Form (role, user), Combobox/Select (role, designation, department, business unit references), MultiSelect/Command (permission picker, direct user permissions), Switch (status, SoD toggles), Badge (isSystem, isAdminOnly, status, rank), Tooltip (permission description), Toast.
- **Acceptance criteria:** (1) the effective-permissions panel computes the deduplicated union of role + direct user grant + designation + business-unit, plus the `all` short-circuit (no priority ordering); (2) a permission change applies on the next request (no caching); re-resolving shows the change immediately, matching the configuration model; (3) `email` is enforced unique case-insensitive and lowercased on save; (4) `isAdminOnly` permissions (e.g. `all`) cannot be granted by a non-super-admin (the cell is disabled with a reason); (5) system roles (SystemAdmin, User) cannot be deleted (`isSystem==true`); (6) deactivating a user removes them from approver/assignee eligibility and forces logout on the next request; (7) `approvalLimit` of 0 stays 0 (approves nothing) and null reads as 200000.
- **Tags:** [OE5 | RA auth-rbac + role-permission-matrix + role-model SoD]

### S01.3 Approval Chain and Routing Rules (archetype: list / form per rule)
- **Route:** `/admin/routing`, with a chain-template panel and a routing-rules table.
- **Roles:** Administrator (act: `approval.configureChain`, `routingRules.update`).
- **Realizes:** configuration step 13: define the ordered approval stages, per-stage type and auto-approval enablement, site-conditional stages, and the routing-rule entries (assignee/approver pools, per-approver limits, default limit). These rows are what the approval engine (flow 03) resolves a chain from.
- **Purpose:** prove the approval chain is configuration, not code: Al Bahja all-to-one and Raphe multi-stage are both just different sets of these rows.
- **Data shown:**
  - **Chain template panel:** the fixed-order stages REQ_DEPARTMENT -> PROCUREMENT -> BUDGET -> FINANCE -> MANAGEMENT, each individually enable-able, with stageType {Standard, Budget, Award, SupplierApproval, Artwork, LanguageReview}, `allowAutoApproval` (per stage), and the site-conditional flag (the Factory Manager rule). The tenant `defaultApprovalLimit` (default 200000) shows here.
  - **Routing rules table (`RoutingRule`):** `ruleId`, `departmentId`, `stage`, `category` (scope, nullable = all), `recordType` (Requisition/PurchaseOrder/Award/Supplier/PaymentSchedule), `valueBandMin` / `valueBandMax` (base currency), `assigneeUserIds` (pool; empty for FINANCE/MANAGEMENT, approval-only), `approverUserIds` (pool), `approverLimits` (map userId -> limit), `allowAutoApproval`, `requiredDesignationRank`, `siteCondition`, `isActive`. Unique per (`departmentId`, `stage`).
- **Primary actions:**
  - "Add routing rule" / "Edit routing rule" (Sheet; perm `routingRules.update`). Set department, stage, category/recordType/value-band scope, the assignee and approver pools (Combobox of active users with `approval.approve` for the approver pool), per-approver limits, `allowAutoApproval`, `requiredDesignationRank`, `siteCondition`. Enforces: unique (department, stage); FINANCE/MANAGEMENT leave `assigneeUserIds` empty (approval-only, A5); MANAGEMENT cannot enable auto-approval (terminal, never auto-approves).
  - "Enable / disable stage" on the chain template (perm `approval.configureChain`).
  - "Activate / deactivate rule" (sets `RoutingRule.isActive`; an inactive rule drops out of chain resolution; deactivating a rule that leaves a department/stage uncovered will stall records with "No routing rule").
- **Secondary actions / navigation:** filter rules by department/stage/recordType; search by user in a pool; preview the resolved chain for a sample (department, category, amount) tuple.
- **States to design:** empty ("No routing rules. The completion gate requires at least one per active department/stage."), loading, populated, validation-error (duplicate (department, stage); empty approver pool on a stage; assignee set on FINANCE/MANAGEMENT; auto-approval on MANAGEMENT), permission-denied.
- **Shadcn:** Card (chain template with Switch per stage + Select stageType), DataTable (routing rules), Sheet + Form (rule editor), Combobox/MultiSelect (assignee/approver pools, by `User.name`), Input (limits, value bands, designation rank), Switch (`allowAutoApproval`, `isActive`), Badge (approval-only, auto-on, inactive), Toast, Alert (coverage warning).
- **Acceptance criteria:** (1) FINANCE and MANAGEMENT rules reject an `assigneeUserIds` pool (approval-only, A5) and MANAGEMENT cannot enable auto-approval (terminal); (2) exactly one rule per (department, stage); a duplicate is rejected; (3) an empty `approverUserIds` pool is flagged because chain resolution would raise "No eligible approver" and stall the record; (4) `approverLimits` accepts 0 (stays 0, forces manual review) and a missing entry falls back to the default 200000 (A7); (5) deactivating a rule that leaves a department/stage uncovered warns that records will stall with "No routing rule"; (6) the value-band and limit comparisons are documented as operating on `totalAmountInBase` (A12).
- **Tags:** [OE2 | ISO 8.4 | platform-services 1 + raphe-routing-approval]

### S01.4 Dynamic Field Configuration (archetype: list / form per field)
- **Route:** `/admin/fields`.
- **Roles:** Administrator (act: `fieldConfig.manage`).
- **Realizes:** configuration step 15: define the field config by stage, scope, and purchase type that drives the intake/detail forms and the mandatory-field gate on stage progression.
- **Purpose:** prove the intake and detail forms adapt per category/purchase-type without code changes, and that the mandatory gate is data-driven.
- **Data shown (`FieldConfig`):** `fieldConfigId`, `fieldName` (key), `label`, `dataType` (string/number/decimal/date/boolean/reference/enum/attachment), `stage` (INITIATION/ORDERED/PARTIAL_DELIVERY/POST_DELIVERY), `scope` (TICKET_LEVEL/ITEM_LEVEL/BLOCK_LEVEL), `purchaseType` (subset of {LOCAL, IMPORT}), `mandatory`, `isAuto`, `showHistory`, `referenceEntity` (when reference: project/vertical/user/supplier/item/ticket-item). Grouped by stage, then scope.
- **Primary actions:**
  - "Add field" / "Edit field" (Sheet; perm `fieldConfig.manage`). Set `fieldName` (unique within stage+scope), `label`, `dataType`, `stage`, `scope`, `purchaseType`, `mandatory`, `isAuto`, `showHistory`, and `referenceEntity` (required when `dataType==reference`).
  - "Toggle mandatory" / "Toggle auto" (sets `FieldConfig.mandatory` / `isAuto`; a mandatory non-auto field gates advance out of its stage; an auto field never gates even if mandatory, A2).
  - "Set purchase-type scope" (restricts a field to LOCAL, IMPORT, or both; import-only fields like HS code / incoterm apply only to import).
- **Secondary actions / navigation:** filter by stage/scope/purchaseType; search by field key/label; preview the rendered INITIATION form for a chosen (category, purchaseType) to confirm which fields show and which gate.
- **States to design:** empty, loading, populated, validation-error (duplicate `fieldName` within stage+scope; `referenceEntity` missing on a reference field), permission-denied.
- **Shadcn:** DataTable (grouped by stage/scope), Sheet + Form (field editor), Select (dataType, stage, scope, referenceEntity), MultiSelect (purchaseType), Switch (mandatory, isAuto, showHistory), Badge (mandatory, auto, import-only), Toast, a preview Card (the rendered form).
- **Acceptance criteria:** (1) a mandatory non-auto field blocks stage advance until filled (per the isEmpty rule: null/blank/[]/{}/invalid-date are empty, 0 and false are present), an auto field never gates even when mandatory (A2); the preview demonstrates both; (2) an import-only field (`purchaseType==[IMPORT]`) appears on the import preview and is hidden/ignored on the local preview; (3) `referenceEntity` is required when `dataType==reference` and is enforced on save; (4) `fieldName` is unique within (stage, scope); (5) field config changes drive the intake form (S02.2) without a code change; the preview reflects edits immediately.
- **Tags:** [OE2 | ISO 7.5 | platform-services 2 + raphe-stage-progression]

### Inline behaviour shown on these screens (not separate screens)
- **Bulk import all-or-nothing (e06):** shown on S01.1 as the upload Dialog's result: a clean file upserts in one transaction; any row error rejects the whole file with a {rowNumber, reason} list and writes nothing (no partial state to clean up).
- **Effective-permission resolution (no caching):** shown on S01.2 as the per-user effective-permissions panel; a grant change re-resolves on the next request.
- **Completion gate readiness:** the Administrator hub (S00.12) reads from these screens: base currency (S01.1), at least one routing rule per active department/stage (S01.3), at least one budget per active cost-center for the period (S01.1 Project/Budget), field config loaded (S01.4), a user per required approval stage (S01.2/S01.3).
