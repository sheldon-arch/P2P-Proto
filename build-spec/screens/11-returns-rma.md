# Screens: 11 Returns and RMA (Build-real)

Flow 11 realized as screens. Source: `documentation/11-returns-rma.md` and `documentation/e04-ncr-capa.md`, data dictionary `Return.json` / `NCR.json` / `CorrectiveAction.json` / `CreditDebitNote.json` / `GrIrEntry.json`, `model/role-permission-matrix.md`, state machines `returnLifecycle` / `ncrLifecycle`. Covers the returns list, the RMA flow (initiate -> authorize -> identify condition -> schedule shipment -> close with credit/debit note), and the NCR/CAPA workbench the return links to.

### S11.1 Returns List (archetype: list)
- **Route:** `/returns`
- **Roles:** Buyer (handle, act: `return.initiate`, `return.authorize`, `return.view`); Quality (initiate on NCR, `return.initiate` conditional NCR); Receiving (receive/schedule, `return.schedule`); Finance-Maker (close with note, `return.close`); others per matrix view.
- **Realizes:** entry point; the open returns and their stage across the RMA lifecycle.
- **Purpose:** the handler's view of every return - where each sits (raised / authorized / blocked / in-transit / closed), whether it is NCR-linked or buyer-initiated, and the financial direction at closure. Answers "what returns are open and what do I act on" quickly.
- **Data shown:** per return from `Return.json` - rmaNumber, sourceOrderId, supplierId, linkedNcrId (NCR-linked vs buyer-initiated), reason (Badge), authorizationStatus (Badge), closureStatus (Badge: raised/authorized/blocked/in-transit/closed), quantityToReturn, creditDebitNoteId (the note at closure).
- **Primary actions:** "Initiate return" (-> S11.2; `return.initiate`); row click -> RMA detail (S11.2-S11.5).
- **Secondary actions / navigation:** faceted filter by closureStatus / reason / NCR-linked / supplier; search by RMA / PO / supplier; link to the source NCR (S11.6) and the source PO (flow 05).
- **States to design:** empty ("No returns. Returns appear from an NCR disposition or a buyer-initiated return."), loading (skeleton), populated, error.
- **Shadcn components:** DataTable, Popover+Command (faceted filter), Input (search), Badge (reason, authorizationStatus, closureStatus, NCR-linked), Button, Toast.
- **Acceptance criteria (believable done):** (1) closureStatus Badge uses the lifecycle values (raised/authorized/blocked/in-transit/closed) and reflects only legal positions; (2) NCR-linked returns show the linkedNcrId and buyer-initiated returns show it null; (3) the reason Badge drives the financial direction shown at closure (supplier-fault -> credit note; over-delivery/wrong-item -> debit note); (4) the list scopes row actions by the matrix (only the routed role acts at each stage).
- **Tags:** [S4 | build-new + AB NCR]

### S11.2 Initiate Return / RMA (archetype: form / wizard step)
- **Route:** `/returns/new`, `/returns/:id`
- **Roles:** Quality (NCR-triggered initiate, act: `return.initiate`); Buyer (buyer-initiated, `return.initiate`).
- **Realizes:** step 2 (S4.1 initiate source return).
- **Purpose:** create the Return/RMA record from an NCR disposition = return or a buyer-initiated reason (over-delivery, wrong item, supplier-acknowledged error), with the lines and reason that will drive the rest of the flow.
- **Data shown / captured:** `Return.json` - rmaNumber (auto, immutable), sourceOrderId (PO; resolves to the order the goods were received against), supplierId (auto from PO), linkedNcrId (mandatory if NCR-triggered, null if buyer-initiated), reason (Select, ReturnReason {defective, damaged, wrong-item, over-delivery, expired, quality-fail}), quantityToReturn (> 0 and <= received quantity for the line), returnType (full-lot / partial-lot / single-line). On submit closureStatus = raised.
- **Primary actions:** "Initiate" (creates the Return; closureStatus = raised; `return.initiate`).
- **Secondary actions / navigation:** add return lines (poLineId, itemId, quantityToReturn, reasonCode); link to the source NCR; percentNonConformance copied from the linked NCR when present.
- **States to design:** empty (new), loading (edit fetch), validation-error (quantityToReturn > received, missing linkedNcrId on NCR-triggered), raised toast.
- **Shadcn components:** Form (react-hook-form + Zod from `Return.json`), Select (reason, returnType), Combobox (sourceOrderId), Sheet (return lines), Badge, Button, Toast.
- **Acceptance criteria (believable done):** (1) rmaNumber is generated once and immutable for the life of the return; (2) linkedNcrId is mandatory for an NCR-triggered return and null for a buyer-initiated one, and must resolve to an existing NCR when set; (3) quantityToReturn must be > 0 and <= the received quantity for the line (a partial return is pro-rata at closure); (4) the reason determines the financial direction (supplier-fault reasons expect a credit note, over-delivery/wrong-item raise a debit note); (5) closureStatus = raised on submit.
- **Tags:** [S4.1 | ISO 8.7 | data-model Return/RMA]

### S11.3 Authorize Return (archetype: form / modal)
- **Route:** `/returns/:id/authorize`
- **Roles:** Buyer (act: `return.authorize`); Supplier (authorizes via the email + OTP external form - see `supplier-portal.md`).
- **Realizes:** steps 3-5 (S4.2 request/authorize, authorizationStatus gate, escalate declined return).
- **Purpose:** request authorization from the supplier and record the supplier-issued RMA reference; no goods move before an authorized RMA number exists.
- **Data shown / captured:** `Return.json` - authorizationStatus (Select {requested, authorized, declined}), supplierRmaNumber (the supplier-issued reference, mandatory before shipment), supplierReturnAddress, restockingFeePercent (default 0). On decline a declineReason and closureStatus = blocked.
- **Primary actions:** "Request authorization" (sends the request to the supplier via the external form; authorizationStatus = requested; `return.authorize`); "Record authorization" (`returnLifecycle` authorize; guard RMA number issued; authorizationStatus = authorized, closureStatus = authorized; `return.authorize`); "Escalate declined" (authorizationStatus = declined, closureStatus = blocked; routes back to the e04 NCR handler for an alternative disposition; a unilateral debit note may still be raised).
- **Secondary actions / navigation:** view the supplier response; link back to the NCR for re-disposition on decline.
- **States to design:** loading, populated, requested (awaiting supplier), authorized (proceeds to S11.4), blocked (declined or no-response after the reminder SLA), escalated toast.
- **Shadcn components:** Form, Select (authorizationStatus), Input (supplierRmaNumber), Dialog (escalate with declineReason), Badge, Button, Toast.
- **Acceptance criteria (believable done):** (1) the S11.4 shipment step is gated on authorizationStatus = authorized AND supplierRmaNumber set - no goods dispatch before an authorized supplier RMA number exists; (2) a declined or no-response-after-SLA authorization sets closureStatus = blocked and routes back to e04 for an alternative disposition, and the underlying NCR is not closed by a declined return; (3) a unilateral debit note may still be raised on a declined return (buyer recovery); (4) restockingFeePercent reduces the expected credit accordingly.
- **Tags:** [S4.2 | ISO 8.7 | build-new + RA OTP form]

### S11.4 Identify Condition + Schedule Shipment (archetype: form / wizard step)
- **Route:** `/returns/:id/condition`, `/returns/:id/shipment`
- **Roles:** Quality (identify condition, act: `return.view` + condition classification); Receiving/Warehouse (schedule shipment, `return.schedule`).
- **Realizes:** steps 6-8 (S4.3 identify product condition/reason, S4.4 schedule product shipment, supplier confirms receipt).
- **Purpose:** classify the physical/functional condition behind the reason, then arrange the return shipment (carrier, label, who-pays, pickup date, tracking, incoterm) and segregate the goods until dispatch.
- **Data shown / captured:** condition from `Return.json` - reason (read), productCondition (max 2000), percentNonConformance (0-100, copied from the linked NCR if present), images (>= 1 for defective/damaged/quality-fail). Shipment from `Return.json` - carrier, returnLabelRef (label reference + who-pays {supplier-paid, buyer-paid, shared}, default supplier-paid for defective/quality-fail), shipmentDate (pickup, >= today), returnTrackingNumber, returnIncoterm (Incoterms 2020; validated against transport mode: FOB/CIF/CFR sea-only; air/multimodal FCA/CPT/CIP/DAP), segregationLocation (quarantine bin reference).
- **Primary actions:** "Identify condition" (`returnLifecycle` identifyCondition; guard reason in ReturnReason; Quality); "Schedule shipment" (`returnLifecycle` scheduleShipment; guard carrier + label + date; sets closureStatus = in-transit, arms the ETA-approaching alarm; Receiving, `return.schedule`).
- **Secondary actions / navigation:** attach condition images; record the supplier receipt confirmation (supplierReceiptDate, supplierReceiptCondition, creditNoteIssued, creditNoteRef) from the supplier external form; link to closure (S11.5).
- **States to design:** loading, populated, validation-error (images missing for defective/damaged/quality-fail, incoterm/mode mismatch, pickup date in the past), condition-identified toast, scheduled (in-transit) toast.
- **Shadcn components:** Stepper (condition -> shipment), Form, Select (who-pays, returnIncoterm), DatePicker (shipmentDate), file Input (images, label), Badge (in-transit), Button, Toast.
- **Acceptance criteria (believable done):** (1) shipment cannot be scheduled until an authorized RMA number exists (the S4.4 step is gated on it); (2) at least one image is mandatory for defective/damaged/quality-fail conditions; (3) returnIncoterm is validated against transport mode (FOB/CIF/CFR sea-only) for consistency on the return leg; (4) scheduling sets closureStatus = in-transit and arms the ETA-approaching alarm; (5) goods are recorded as segregated (segregationLocation) from raising until dispatch (ISO 8.7).
- **Tags:** [S4.3/S4.4 | ISO 8.7 | build-new + AB delivery tracking]

### S11.5 Close / Adjust Return (archetype: form / modal)
- **Route:** `/returns/:id/close`
- **Roles:** Finance-Maker (act: `return.close`, `creditNote.manage`, `debitNote.manage`); Finance-Checker (releases any debit-note payment adjustment, maker/checker control).
- **Realizes:** steps 9-13 (financial adjustment direction, apply credit note, raise debit note, S4.5 close/adjust, update NCR + feed scorecard).
- **Purpose:** close the return by adjusting the payable in the correct direction - a supplier credit note (supplier fault) or a buyer debit note (buyer recovery) - or reversing the GR/IR accrual alone when nothing was invoiced; always reversing GR/IR and reconciling the commitment for the returned quantity.
- **Data shown / captured:** `Return.json` - creditDebitNoteId (set at closure; null for accrual-only). The note from `CreditDebitNote.json` - type (credit / debit), supplierId, linkedReturnId (auto), linkedInvoiceId, amount (returned value; + supplier-paid return freight on a debit note where applicable), taxAmount (reverses recoverable input tax on the returned portion), reason (auto from reason). The reversal from `GrIrEntry.json` (openAmount reversed pro-rata to quantityToReturn) and the commitment reconciliation.
- **Primary actions:** "Close with credit note" (Branch A: reason in {defective, damaged, expired, quality-fail} AND value already invoiced; posts a supplier CreditNote, reduces the payable, reverses GR/IR pro-rata; `creditNote.manage`); "Close with debit note" (Branch B: reason in {over-delivery, wrong-item} OR supplier disputes credit; raises a buyer DebitNote, reduces payable / raises receivable, maker/checker control; `debitNote.manage`); "Close (accrual-only)" (Branch C: goods not yet invoiced; reverses the GR/IR accrual only, no note, commitment relief only). All set `returnLifecycle` closeOrAdjust -> closureStatus = closed; adjust the creditor ledger; update the linked NCR; feed the supplier scorecard.
- **Secondary actions / navigation:** view the GR/IR reversal; view the creditor-ledger posting; on closure mark the NCR disposition return executed and hand systemic root-cause to the CAPA loop (S11.6).
- **States to design:** loading, populated (branch chosen by reason + invoice state), credit-note sub-form, debit-note sub-form (maker/checker), accrual-only (no note), closed toast.
- **Shadcn components:** multi-pane (return detail + note), Select (note type, driven by reason), Sheet (credit/debit-note sub-form), Badge (closed), Button, Toast.
- **Acceptance criteria (believable done):** (1) the reason determines the financial direction - supplier-fault (defective/damaged/expired/quality-fail) posts a supplier credit note, buyer recovery (over-delivery/wrong-item) or a disputed credit raises a buyer debit note, and goods-not-yet-invoiced reverses GR/IR only with no note; (2) closure always reverses the GR/IR accrual pro-rata to quantityToReturn and reconciles the commitment for the returned quantity; (3) a debit note follows the maker/checker financial control (Finance-Checker releases the adjustment); (4) the note posts against the per-supplier creditor ledger and the tax on the returned portion is reversed; (5) closure updates the linked NCR (marks the return disposition executed) and feeds the supplier scorecard (quality + four-factor perfect-order delivery dimensions).
- **Tags:** [S4.5 | ISO 8.7 -> 8.4.1 | build-new credit/debit note]

### S11.6 NCR / CAPA Workbench (archetype: workbench)
- **Route:** `/quality/ncr/:id`, `/quality/capa/:id`
- **Roles:** Quality (act: `ncr.raise`, `qc.reject`, `capa.manage`, effectiveness review); Buyer (issue SCAR, resolve with supplier, `capa.manage`); Supplier (root-cause + corrective action via the external form); Engineering (raise NCR on the spares path, `ncr.raise` conditional).
- **Realizes:** the e04 loop steps 5-16 (raise NCR, disposition, return to RMA, SCAR/CAPA, supplier root-cause, effectiveness review, feed re-evaluation, suspend gate).
- **Purpose:** the differentiated screen - the closed quality loop where an NCR is dispositioned, a systemic cause opens a CAPA/SCAR to the supplier, effectiveness is reviewed, the result feeds the supplier re-evaluation, and the supplier may trip toward SUSPENDED. The return (S11.2-S11.5) is the disposition = return branch off this workbench.
- **Data shown / captured:** `NCR.json` - id, itemId, supplierId, deliveryNoteNumber, deliveryDate, description (max 2000), percentNonConformance (0-100), image (>= 1 for defective/damaged/quality-fail), disposition (Select {return, rework, use-as-is-concession, scrap}), decidingAuthority, status (raised/dispositioned/closed). `CorrectiveAction.json` - identifier, ncrId, supplierId, rootCause (5-why), action, preventiveAction, targetDate, evidence, effectivenessReview (Select {pending, effective, not-effective}), isSCAR, status (open/action-proposed/under-review/closed/re-opened). The supplier's consecutive-below streak and current AVL status from `SupplierScorecard.json`.
- **Primary actions:** "Disposition NCR" (`ncrLifecycle` disposition; disposition = return hands to the RMA (S11.2) carrying linkedNcrId; rework loops to re-inspection; concession requires documented authority; scrap writes off; Quality); "Raise CAPA / SCAR" (`ncrLifecycle` raiseCapa; opens a CorrectiveAction with a response SLA to the supplier; Buyer, `capa.manage`); "Submit root-cause" (Supplier, via external form; CAPA status = action-proposed); "Review effectiveness" (`ncrLifecycle` closeCapa on effective; effectivenessReview = effective closes the CAPA and NCR and feeds re-evaluation; not-effective re-opens the SCAR; Quality).
- **Secondary actions / navigation:** view the supplier scorecard (flow 12); the suspend gate (`supplierLifecycle` suspend) on a critical NCR / two consecutive periods below / failed audit / expired cert / sanctions / active not-effective CAPA; link to the return branch (S11.2).
- **States to design:** loading, populated (raised -> dispositioned -> closed), CAPA-open (awaiting supplier root-cause), under-review (effectiveness), re-opened (not-effective loop), NCR-closed toast, suspend toast.
- **Shadcn components:** multi-pane (NCR + CAPA + supplier history), Select (disposition, effectivenessReview), Textarea (rootCause, action, preventiveAction), file Input (evidence, image), Badge (NCR status, CAPA status, AVL status, consecutive-below streak), Button, Toast.
- **Acceptance criteria (believable done):** (1) disposition = return hands off to the RMA flow (S11.2-S11.5) carrying linkedNcrId, running in parallel with the systemic CAPA; (2) a not-effective effectiveness review re-opens the SCAR (the loop) and feeds the consecutive-below counter, while effective closes the CAPA and the source NCR and feeds the re-evaluation; (3) the CAPA close feeds the supplier scorecard re-evaluation (8.4.1) and may trip the supplier toward SUSPENDED on a critical NCR / two consecutive periods below / active not-effective CAPA (the CAPA-driven grade drop and suspend gate, axiom A11); (4) a SUSPENDED supplier is blocked from new POs (supplierLifecycle suspend blocks new PO issue); (5) an open NCR or unresolved CAPA blocks the requisition reaching COMPLETED (axiom A3).
- **Tags:** [S2.5/S4 | ISO 8.6 -> 8.7 -> 10.2 -> 8.4.1 | e04 NCR/CAPA + build-new CAPA]

### Inline edge behaviours shown on these screens (not separate screens)
- **NCR link (A11):** shown inline on S11.1/S11.2 as the linkedNcrId; the return disposition originates in the NCR workbench (S11.6) and the closed return feeds back to it.
- **No-RMA hard gate (S4.2):** shown inline on S11.3/S11.4 - the shipment step is disabled until authorizationStatus = authorized and supplierRmaNumber is set.
- **Pro-rata reversal:** shown inline on S11.5 - the credit/debit note and GR/IR reversal are pro-rata to quantityToReturn; a restockingFeePercent reduces the expected credit.
- **Declined return:** shown inline on S11.3 - a declined or no-response return sets closureStatus = blocked and re-disposition routes back to S11.6.
- **CAPA-driven grade drop / suspend:** shown inline on S11.6 as the consecutive-below streak and AVL status; recurring NCRs auto-affect the composite (feeds flow 12).
