# Screens: 10 Payments and Installments (Build-real)

Flow 10 realized as screens. Source: `documentation/10-payments.md`, data dictionary `Payment.json` / `PaymentSchedule.json` / `Installment.json` / `AdvancePayment.json` / `Retention.json` / `PaymentMilestone.json` / `CashFloat.json` / `CreditorLedger.json`, `model/role-permission-matrix.md`, state machine `installmentLifecycle`. Covers the payment schedule and due breakdown (Finance-Maker landing), installment approve (full vs partial + remainder), process + receipt (PROCESSED terminal), reschedule (captures originalDate), the payment release queue (Finance-Checker landing, maker != checker), advance/retention, the cash-float widget, and the creditor ledger view.

### S10.1 Payment Schedule and Due (archetype: list / per-role landing for Finance-Maker)
- **Route:** `/payments`
- **Roles:** Finance-Maker (act: `payments.schedules.view`, `payments.schedules.approve`, `payments.schedules.process`); Buyer / Fin-Checker / Management (view, `payments.schedules.view`); Supplier (own, conditional, via portal).
- **Realizes:** steps 1-3 (build schedule, schedule-locked guard), 23-25 (next-entry loop, overdue reminder); the other half of the Finance-Maker landing from storyboard section 6 (invoices to match + payment schedule due).
- **Purpose:** the AP maker's payment home - every schedule and its installments with the paid / upcoming / overdue breakdown, so the maker can approve the due installments and prepare releases. Answers "what is due, what is overdue, what can I pay" in the first 30 seconds.
- **Data shown:** per payment from `Payment.json` - identifier, supplierId, poId, invoiceId, totalAmount, totalAmountPaid (computed), upcomingPaymentAmount (computed), overduePaymentAmount (computed); the live amount breakdown (Paid = sum approvedAmount across PROCESSED; Upcoming = non-processed with date >= start-of-today or null; Overdue = non-processed with date < start-of-today). Per installment from `Installment.json` - identifier, amount, dueEvent or date, description, term, status (Badge: pending/APPROVED/PARTIAL_APPROVAL/PROCESSED/RESCHEDULED), approvedAmount; from `PaymentSchedule.json` the locked flag.
- **Primary actions:** "Approve installment" (-> S10.2; `installmentLifecycle` approve; `payments.schedules.approve`); "Process" (-> S10.4; `payments.schedules.process`; enabled only on APPROVED/PARTIAL_APPROVAL); "Reschedule" (-> S10.5; `payments.schedules.reschedule`); row click -> payment detail.
- **Secondary actions / navigation:** faceted filter by status / overdue / supplier / term; search by PO / supplier / payment ref; link to the creditor ledger (S10.7) and cash float (S10.6). The overdue reminder (~28 days before a 30/60/90-day term is due) surfaces as an inline row Badge (fired by System, informational, never changes status).
- **States to design:** empty ("No payment schedules. Schedules appear once an invoice is approved or a PO advance is due."), loading (skeleton), populated, error.
- **Shadcn components:** DataTable (payments + nested installments), Badge (installment status, overdue, locked), Card (the paid/upcoming/overdue KPI strip), Button (Process disabled unless APPROVED/PARTIAL_APPROVAL), Toast.
- **Acceptance criteria (believable done):** (1) the paid/upcoming/overdue breakdown is recomputed live and never cached, with the overdue boundary at 00:00:00 start-of-today and a null installment date treated as upcoming (never overdue); (2) Paid sums approvedAmount across PROCESSED installments only; (3) the Process action is hidden/disabled on any installment not in {APPROVED, PARTIAL_APPROVAL} and is never shown on a PROCESSED installment (PROCESSED is terminal, no un-process); (4) a locked schedule (any installment APPROVED/PARTIAL_APPROVAL/PROCESSED) shows no edit/delete on existing entries; (5) the overdue reminder Badge is informational and does not advance the installment status.
- **Tags:** [S2.7 | RA installments + AB payment terms]

### S10.2 Approve Installment (archetype: form / modal)
- **Route:** `/payments/:paymentId/installments/:id/approve`
- **Roles:** Finance-Maker (act: `payments.schedules.approve`; maker, never the checker).
- **Realizes:** steps 8-12 (approve installment amount-vs-agreed, APPROVED full, PARTIAL_APPROVAL + one remainder, apply withholding, recoup advance + withhold retention).
- **Purpose:** the maker decides how much of an agreed installment to approve; a full approval sets APPROVED, a partial approval sets PARTIAL_APPROVAL and creates exactly one remainder for the leftover, and the System nets withholding / advance recoupment / retention to the release.
- **Data shown / captured:** `Installment.json` - amount (agreed; read), approvedAmount (entered, ApproveInstallmentDto: > 0 and <= agreed), approvalNote, status (set by the System), remainderInstallmentId (set on partial), approvedById/approvedAt (auto). The netting preview from `Payment.json` - withholdingTaxAmount (from `Payment.withholdingTaxCodeId`, cross-border services), advanceRecoupedAmount (from `AdvancePayment.outstandingAmount`), retentionWithheldAmount (from `Retention.withheldPercent`), netReleaseAmount (= approvedAmount - withholding - advance recouped - retention withheld).
- **Primary actions:** "Approve full" (`installmentLifecycle` approve; guard approvedAmount == agreed -> status APPROVED, no remainder; records approvedAmount = agreed; `payments.schedules.approve`); "Approve partial" (`installmentLifecycle` approve; guard 0 < approvedAmount < agreed -> status PARTIAL_APPROVAL + creates exactly one remainder installment with null status, leftover = agreed - approved, same date/schedule/description; `payments.schedules.approve`). An approvedAmount <= 0 or > agreed is rejected (BadRequestException) and the installment stays pending.
- **Secondary actions / navigation:** view the netting breakdown; view the linked advance / retention; back to the schedule.
- **States to design:** loading, populated, validation-error (amount out of range -> BadRequestException, stays pending), full-approval toast, partial-approval toast (showing the created remainder), permission-denied (a Finance-Checker cannot approve here - maker != checker).
- **Shadcn components:** Form, Input (approvedAmount), Textarea (approvalNote), Card (netting preview), Badge (resulting status), Button, Toast.
- **Acceptance criteria (believable done):** (1) approvedAmount must be > 0 and <= agreed, else a BadRequestException and the installment stays pending (the approve validation rule); (2) approvedAmount == agreed sets APPROVED with no remainder, while 0 < approvedAmount < agreed sets PARTIAL_APPROVAL and creates exactly one remainder (leftover = agreed - approved, same date/schedule/description, null status), never more than one; (3) approving any installment locks the schedule (no edit/delete of existing entries thereafter); (4) the net release nets approvedAmount minus withholding minus advance recouped minus retention withheld; (5) the maker who approves is recorded in approvedById, distinct from the checker who releases (maker != checker).
- **Tags:** [S2.7 | RA e04 approve + remainder]

### S10.3 Payment Release Queue (archetype: queue / per-role landing for Finance-Checker)
- **Route:** `/payments/release-queue`
- **Roles:** Finance-Checker (act: `payments.schedules.approve` release; the only second-person release gate); Management (release at/above the configurable threshold, `payments.approve` conditional management).
- **Realizes:** steps 13-15 (maker prepares for release, checker reviews release, approved-to-pay vs hold); the Finance-Checker landing from storyboard section 6 (payments awaiting release).
- **Purpose:** the Chief Accountant's home - releases the maker has prepared, reviewed against the schedule and cash flow, to approve to pay or hold. The financial-control segregation: the maker prepares, the checker releases, never the same person.
- **Data shown:** per prepared release from `Payment.json` - identifier, supplierId, poId, netReleaseAmount, currency; the installment(s) from `Installment.json` (approvedAmount, status APPROVED/PARTIAL_APPROVAL); who prepared it (the maker, `Installment.approvedById`) for the maker != checker check; releases at/above managementReleaseThreshold flagged for Management approval.
- **Primary actions:** "Approve to pay" (checkerDecision = approve; routes to the process guard, then S10.4; `payments.schedules.approve` release; condition checker != the maker who prepared it); "Hold" (checkerDecision = hold; signatory unavailable or cash flow; routes to reschedule S10.5). For releases >= managementReleaseThreshold, "Management approve" precedes the checker release (Management; below threshold the step auto-passes).
- **Secondary actions / navigation:** filter by amount / supplier / above-threshold; view the prepared release detail; view cash flow against the schedule.
- **States to design:** empty ("No payments awaiting release."), loading, populated, permission-denied (the maker who prepared a release sees no release action on it - maker != checker), approved-to-pay toast, hold toast.
- **Shadcn components:** DataTable, Badge (above-threshold, status), Dialog (approve-to-pay / hold with reason), Button (disabled when checker == preparer), Toast.
- **Acceptance criteria (believable done):** (1) the maker who prepared a release cannot release it - the release action is denied for that user (maker != checker, the only second-person release gate, financial-control SoD per SOX/COSO and ISO 37001); (2) Approve to pay proceeds only to the process guard (a payment is still processed by the maker, never the checker); (3) Hold reschedules with reason and re-enters prepare when due; (4) a release at/above the configurable management threshold requires Management approval first, and below it the checker approval suffices and the management step auto-passes; (5) the queue shows only prepared releases (APPROVED/PARTIAL_APPROVAL installments), not pending ones.
- **Tags:** [S2.7 | AB Chief Accountant + financial-control SoD]

### S10.4 Process Payment + Receipt (archetype: form / modal)
- **Route:** `/payments/:paymentId/installments/:id/process`
- **Roles:** Finance-Maker (act: `payments.schedules.process`; the maker processes after the checker releases).
- **Realizes:** steps 16-22 (processable-status guard, process payment + upload receipt, recompute totals + ledgers, destination split, email confirmation, float top-up).
- **Purpose:** the maker disburses the released installment, attaches the receipt, and the System recomputes the breakdown, posts the creditor ledger, and routes the confirmation to the supplier (or tops up the float for a cash reimbursement). PROCESSED is terminal.
- **Data shown / captured:** `Installment.json` (ProcessInstallmentDto, multipart) - paymentDate (entered), processNotes, receiptFile (optional; stored at receipts/{ticketId}/{originalFilename}; an upload failure fails the whole operation); on success status = PROCESSED, processedAt, processedById. From `Payment.json` - netReleaseAmount, destination ({supplier -> email confirmation on the order thread; cashHandler -> float top-up}), realizedFxGainLoss (computed between PO-date and payment-date rate).
- **Primary actions:** "Process payment" (`installmentLifecycle` process; guard status in {APPROVED, PARTIAL_APPROVAL}; sets PROCESSED, processedAt, processedById, optional receipt; recomputes paid/upcoming/overdue, updates the creditor ledger and the MANAGEMENT vertical completion; `payments.schedules.process`).
- **Secondary actions / navigation:** upload the receipt; view the recomputed breakdown; the supplier destination emails the confirmation, the cash-handler destination tops the float back to its ceiling.
- **States to design:** loading, populated, not-processable (the Process action is hidden/disabled unless status in {APPROVED, PARTIAL_APPROVAL}; never shown on PROCESSED), receipt-upload-failure (the whole process fails, status not set), processed toast (terminal, no further Process button).
- **Shadcn components:** Form (multipart), DatePicker (paymentDate), file Input (receipt), Textarea (processNotes), Badge (status), Button, Toast.
- **Acceptance criteria (believable done):** (1) only APPROVED or PARTIAL_APPROVAL installments are processable - processing a pending, RESCHEDULED or already-PROCESSED installment raises a BadRequestException; (2) PROCESSED is terminal with no un-process, and the Process button is gone on a PROCESSED installment; (3) a receipt upload failure fails the whole process operation so a PROCESSED entry is never left without its intended receipt; (4) processing recomputes the paid/upcoming/overdue breakdown live and posts the supplier creditor ledger; (5) the destination split routes a supplier release to an email confirmation on the order thread and a cash-handler release (a float reimbursement) to a float top-up to the ceiling.
- **Tags:** [S2.7 | RA process + receipt path]

### S10.5 Reschedule Installment (archetype: form / modal)
- **Route:** `/payments/:paymentId/installments/:id/reschedule`
- **Roles:** Finance-Checker (act: `payments.schedules.reschedule`); Finance-Maker (`payments.schedules.reschedule` conditional with reason).
- **Realizes:** step 18 (reschedule with reason, capture originalDate).
- **Purpose:** defer a held installment to a new date with a reason, preserving the originally promised date so the deferral is visible and never a silent date change.
- **Data shown / captured:** `Installment.json` - newDate (entered; sets date and RESCHEDULED), rescheduleReason (the hold reason: signatory unavailable / cash flow), originalDate (captured from the current date on the first reschedule, preserved unchanged thereafter), amount (unchanged).
- **Primary actions:** "Reschedule" (`installmentLifecycle` reschedule; sets date = newDate and status RESCHEDULED; the first reschedule captures originalDate; `payments.schedules.reschedule`). A RESCHEDULED entry re-enters prepare when due and can be approved again.
- **Secondary actions / navigation:** view the reschedule history (originalDate vs current date); back to the schedule.
- **States to design:** loading, populated, validation-error (invalid newDate), first-reschedule (originalDate captured), later-reschedule (originalDate and amount unchanged), rescheduled toast.
- **Shadcn components:** Form, DatePicker (newDate), Textarea (rescheduleReason), Badge (RESCHEDULED, originalDate vs current), Button, Toast.
- **Acceptance criteria (believable done):** (1) the first reschedule captures the current date as originalDate and preserves it unchanged on every later reschedule (the original commitment is never lost); (2) reschedule sets status RESCHEDULED and the new date, leaving amount and originalDate unchanged on later reschedules; (3) a RESCHEDULED entry remains editable (only null and RESCHEDULED entries are editable on a locked schedule) and re-enters prepare when due; (4) the rescheduleReason is recorded so the deferral shows in the audit trail, never a silent date change.
- **Tags:** [S2.7 | RA reschedule + AB hold-reschedule]

### S10.6 Cash Float Widget (archetype: card / form)
- **Route:** `/payments/cash-float`
- **Roles:** Buyer (cash buyer, act: `payments.cashFloat.manage` conditional); Finance-Maker (reimburse, `payments.cashFloat.manage` conditional); Finance-Checker (approve reimbursement, `payments.cashFloat.manage` conditional).
- **Realizes:** steps 4-7 (cash float buy no PO, log receipt, float nearing limit, request reimbursement) and steps 20-22 (float top-up).
- **Purpose:** the petty-cash track for local on-the-spot buys with no PO; deduct a cash buy from the float, log the vendor receipt, and request a maker/checker-approved reimbursement when the float runs low. Shown as a compact widget plus a spend log.
- **Data shown / captured:** `CashFloat.json` - identifier, handlerId, floatAmount (ceiling, default OMR 300), balance (current; a buy allowed only when cashPrice <= balance), currency, reimbursementThreshold (default ~1/3 of allotment, ~OMR 100), cashGrnNoPoRef (true; a cash buy raises a GoodsReceipt with poId = null), status (ACTIVE/INACTIVE), lastReimbursedAt. The cash-buy fields - spendAmount (> 0), vendorReceipt (file), reason.
- **Primary actions:** "Log cash buy" (deducts spendAmount from balance, raises a cash GRN with no PO reference, attaches the vendor receipt; Buyer; `payments.cashFloat.manage`; condition spendAmount <= balance); "Request reimbursement" (when balance <= reimbursementThreshold; routes into the maker-prepares pipeline with the cash handler as the destination; tops the float back to the ceiling on checker approval; `payments.cashFloat.manage`).
- **Secondary actions / navigation:** view the cash spend log (reported alongside credit spend); the cash GRN with poId = null still applies QC on QC-controlled items.
- **States to design:** loading, populated (float widget + spend log), low-water (balance <= threshold, the reimbursement request surfaced), buy-blocked (spendAmount > balance), reimbursed toast (float restored to ceiling), INACTIVE float (no buys).
- **Shadcn components:** Card (float balance gauge), Form (cash buy), file Input (vendorReceipt), DataTable (spend log), Badge (low-water, status), Button, Toast.
- **Acceptance criteria (believable done):** (1) a cash buy is allowed only when spendAmount <= balance and raises a GoodsReceipt with poId = null (the no-PO cash GRN, the structural marker of the cash branch); (2) when balance <= reimbursementThreshold (~1/3 of the allotment) the reimbursement request is surfaced and runs through the maker/checker pipeline (maker prepares, checker approves) with the cash handler (not a supplier) as the destination; (3) reimbursement restores the balance to floatAmount (the ceiling); (4) cash spend is tagged and reported alongside credit spend (normalized to base via FX); (5) QC still applies the COA hard block on a QC-controlled item bought on the float.
- **Tags:** [S2.7 | AB cash float + cash GRN]

### S10.7 Creditor Ledger (archetype: detail / list)
- **Route:** `/payments/creditors/:supplierId`
- **Roles:** Buyer / Finance-Maker / Finance-Checker / Management (view, `creditorLedger.view`); read-only to procurement.
- **Realizes:** step 19 (update the per-supplier creditor ledger).
- **Purpose:** the per-supplier payables view - the running amount owed, the supplier terms, and the append-only transaction history of invoices, payments, credit/debit notes, advances and retentions; the aging/DPO pivot. Read-only, drawn from existing transactional data.
- **Data shown:** `CreditorLedger.json` - supplierId, supplierName, currency, amount (running balance owed), paymentTerms (default net 90), overdueAmount (past-due, boundary start-of-today), lastUpdatedAt, transactionHistory (append-only, timestamped entries {type, refId, amount, date, balanceAfter}). Advances and retentions outstanding from `AdvancePayment.outstandingAmount` / `Retention.withheldAmount`.
- **Primary actions:** none (read-only to procurement; the ledger is posted to by the payment flow, never hand-entered).
- **Secondary actions / navigation:** filter the transaction history by type / date; aging buckets; link to the source invoice / payment / note; DPO and creditor-aging surface here for analytics (flow 12).
- **States to design:** empty (no transactions yet for the supplier), loading, populated, error.
- **Shadcn components:** Card (balance + terms + overdue), DataTable (transaction history), Badge (overdue, transaction type), no action buttons.
- **Acceptance criteria (believable done):** (1) the view is read-only to procurement (name, amount, payment terms, transaction history) and is drawn from existing transactions, never hand-entered; (2) the transactionHistory is append-only, timestamped and never edited or deleted; (3) the running balance increases on approved payables / debit notes and decreases on payments / credit notes / advance recoupment / retention withheld; (4) overdueAmount uses the start-of-today boundary, consistent with the payment breakdown; (5) balances carry a base-currency sibling so cross-currency aging is comparable.
- **Tags:** [S2.7 | AB creditor ledger]

### Inline edge behaviours shown on these screens (not separate screens)
- **Partial approval remainder (A9):** shown inline on S10.2 - a partial approval creates exactly one remainder (leftover = agreed - approved, same date/schedule/description, null status) that re-enters the next-entry loop.
- **Locked-schedule rule (A9):** shown inline on S10.1 - once any installment is APPROVED/PARTIAL_APPROVAL/PROCESSED the schedule is locked, edits/deletes of existing entries are silently skipped, and only null/RESCHEDULED entries remain editable (new schedules are always created).
- **PROCESSED terminal (A9):** shown inline on S10.1/S10.4 - the Process button is hidden on a PROCESSED installment; PROCESSED has no un-process.
- **Maker != checker (A6):** shown inline on S10.3 - the maker who prepared a release sees no release action on it; the checker releases, the maker processes.
- **Financial-info revert (A7):** shown inline - editing payTerms or paymentSchedules reverts auto-approved Finance/Management completions to AWAITING_APPROVAL (from `PaymentSchedule.financialRevertOnEdit`).
- **Withholding / advance / retention netting:** shown inline on S10.2/S10.4 as the netting preview (net = approvedAmount - withholding - advance recouped - retention withheld).
- **Overdue reminder (~28d):** shown inline on S10.1 as a row Badge; fired by the System about 28 days before a 30/60/90-day term is due; informational, never changes status.
