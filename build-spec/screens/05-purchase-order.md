# Screens: 05 Purchase Order (Build-real)

Flow 05 realized as screens. Source: `documentation/05-purchase-order.md`, data dictionary `PurchaseOrder.json` / `POLine.json` / `FrameworkAgreement.json` / `CallOffRelease.json` / `Supplier.json` / `TaxCode.json` / `Budget.json` / `Commitment.json`, `model/role-permission-matrix.md`, state machine `poLifecycle` (DRAFT -> ISSUED -> ACKNOWLEDGED -> CLOSED; plus DRAFT -> CANCELLED) and the supplier-acknowledge / amend transitions.

### S05.1 Purchase Orders (archetype: list)
- **Route:** `/purchase-orders`
- **Roles:** Procurement/Buyer (act: create, edit, amend); Approver (view, act on PO approval); Budget Owner (view, over-budget overrides); Supplier (portal, own POs only); Management / Finance (view); Administrator (view all).
- **Realizes:** entry point to ordering; the "POs to issue" portion of the Buyer landing (storyboard section 6).
- **Purpose:** the buyer's view of all orders and their lifecycle position, plus the entry to draft a new PO from an awarded quote or release a framework call-off as a PO.
- **Data shown:** `PurchaseOrder.number`, `PurchaseOrder.supplierName` + `supplierCode`, `PurchaseOrder.type` (Badge: item | service | maintenance | freight), `PurchaseOrder.value` + `currency`, `PurchaseOrder.valueInBase`, `PurchaseOrder.deliveryDate`, `PurchaseOrder.status` (Badge per `poLifecycle`: draft / awaiting-approval / issued / acknowledged / closed / cancelled), `PurchaseOrder.acknowledged`, `PurchaseOrder.frameworkAgreementId` (call-off marker).
- **Primary actions:** "New PO" (-> S05.2; perm `po.create`; from an awarded quote / approved requisition); row click -> PO detail (S05.3).
- **Secondary actions / navigation:** faceted filter by status / type / supplier / currency; search by PO number / supplier / requisition; sort by delivery date; pagination (default 10, max 50); link to the awarding contract (S04.5) and requisition (S02.3).
- **States to design:** empty ("No purchase orders yet"), loading (skeleton table), populated, error, permission-denied.
- **Shadcn components:** DataTable, Popover + Command (faceted filter), Badge (status / type / acknowledged), Input (search), Button.
- **Acceptance criteria (believable done):** (1) `PurchaseOrder.status` Badges use the canonical `poLifecycle` states and only the legal next action is offered per row; (2) a call-off PO (`frameworkAgreementId` set) is visibly marked; (3) "New PO" is gated by `po.create` and hidden for non-buyers; (4) rows are scoped per role (a supplier reaching the portal list sees only its own POs); (5) empty state shows for a fresh persona, never a blank table.
- **Tags:** [S2.1 | AB + RA]

### S05.2 Create / Issue Purchase Order (archetype: form / wizard)
- **Route:** `/purchase-orders/new?requisitionId=:id&quotationId=:qid`, `/purchase-orders/:id/edit`
- **Roles:** Procurement/Buyer (act: `po.create`, `po.edit`, `po.issue`; cannot approve own PO); Budget Owner (act on the over-budget override inline); Tax/Compliance (conditional, enter tax on lines).
- **Realizes:** narrative steps 3 (draft PO from requisition + awarded quote), 4 (enter lines, tax, tolerance), 5-7 (budget hard-commit gate + over-budget override + encumber), 8 (submit for approval, SoD), 11-12 (issue, generate number, render and email PDF).
- **Purpose:** assemble a committed buying instruction from the awarded quote: header, lines with tax and tolerance, the hard budget commitment, the SoD approval, and the PDF issue to the supplier.
- **Data shown / captured:** HEADER from `PurchaseOrder.json` - `PurchaseOrder.number` (auto on issue, read-only, "pending until issued" before), `PurchaseOrder.requisitionId` / `quotationId` / `supplierId` / `supplierCode` (read-only, from award), `PurchaseOrder.type` (Select: item | service | maintenance | freight; defaults from requisition category; drives the dynamic form and the match path), `PurchaseOrder.currency` (Select, defaults from quotation), `PurchaseOrder.exchangeRate` (decimal, optional per-PO override, else FX service rate), `PurchaseOrder.incoterm` (Select, Incoterm enum, defaults from quotation; validated vs transport mode), `PurchaseOrder.scopeOfWork` (textarea, mandatory for service/maintenance), `PurchaseOrder.termsAndConditions` (textarea, from tenant template), `PurchaseOrder.deliveryPlace` (Warehouse/site), `PurchaseOrder.deliveryDate` (date, >= today, defaults from needDate), `PurchaseOrder.contactPerson`, `PurchaseOrder.isoDocRef` (optional), `PurchaseOrder.attachments`. LINES from `POLine.json` in a Sheet/Table - `POLine.itemId` (ONBOARDED item or auto-created; free-text for service/maintenance), `POLine.description` (from item, editable), `POLine.quantity` (> 0), `POLine.unitOfMeasure` (Select, UoM), `POLine.agreedPrice` (>= 0, defaults from awarded quote line), `POLine.tolerancePercent` (optional, +/- 5 to 10 for labels/cartons), `POLine.taxCodeId` (Select, TaxCode master, mandatory where jurisdiction requires; reverse-charge for imports), computed `POLine.taxableAmount` / `taxAmount` / `lineValue`. TOTALS: computed `PurchaseOrder.value` and `valueInBase`. OVER-BUDGET OVERRIDE block (Budget Owner): decision (approve-override | reject), overrideReason (mandatory on approve-override), read-only overrideAmount (= valueInBase - availableAmount) from `Budget.availableAmount`.
- **Primary actions:** "Save Draft" (`PurchaseOrder.status` = draft); "Submit for approval" (`poLifecycle` implied draft -> awaiting-approval; routed to a PO approver who is NOT the creator, SoD); "Issue PO" (`poLifecycle` DRAFT -> ISSUED; perm `po.issue`; guard: budget committed (axiom A4) AND supplier ONBOARDED AND buyer != approver (axiom A6); effect: hard-commit/encumber budget, generate `PurchaseOrder.number`, render and store the PO PDF, email it to the supplier); "Emit as PDF" (render and download/store the PO PDF). Budget Owner "Approve override" / "Reject" on the over-budget path.
- **Secondary actions / navigation:** add line (Sheet), remove line, duplicate line, attach file; link to the requisition (S02.3), the awarding contract (S04.5); the incoterm-vs-mode and FX-degradation inline checks (see inline behaviours).
- **States to design:** new (header pre-filled from award, lines pre-filled from awarded quote), validation-error (incoterm invalid for transport mode, delivery date past, missing tax code where required), over-budget (override block shown, issue blocked until override approved), awaiting-approval, issued toast/redirect (with PDF emitted), permission-denied (buyer attempting to approve own PO).
- **Shadcn components:** Stepper or Form with Card sections, Select (type, currency, incoterm, tax code, UoM, warehouse), DatePicker (delivery date), Sheet (line add/edit), Input/Textarea, file Input, Dialog (issue confirmation, over-budget override), Badge (status), Toast.
- **Acceptance criteria (believable done):** (1) "Issue PO" is gated so it cannot fire unless budget is hard-committed (axiom A4), the supplier is ONBOARDED (`Supplier.status`), and the issuing approver is NOT the creating buyer (SoD, axiom A6) - a buyer cannot approve/issue their own PO and that action does not render for the creator; (2) when `Budget.availableAmount` < `PurchaseOrder.valueInBase` the over-budget override block appears and issue is BLOCKED until the Budget Owner approves the override with a mandatory `overrideReason` (logged); (3) `PurchaseOrder.incoterm` is validated against the transport mode (FOB/CFR/CIF sea-only) and an invalid combination is rejected inline; (4) tax is a first-class line attribute - `POLine.taxCodeId` drives computed `taxAmount` and `lineValue`, and reverse-charge applies for imports, so the line totals include tax; (5) `PurchaseOrder.type` drives the dynamic form (service/maintenance require `scopeOfWork` and use the two-way / milestone match path, no GRN; item uses three-way); (6) on issue, `PurchaseOrder.number` is generated transactionally from the configurable pattern and becomes immutable, the PDF is rendered/stored, and budget is encumbered (commitment created, `Budget.availableAmount` reduced, committedAmount increased).
- **Tags:** [S2.1 / OE2 / OE11 | ISO 8.4.3 | AB PO form + SoD + budget hard-commit + incoterm-vs-mode]

### S05.3 Purchase Order Detail (archetype: detail)
- **Route:** `/purchase-orders/:id`
- **Roles:** Procurement/Buyer (act: amend pre-receipt); Approver (act: approve/reject if awaiting-approval); Supplier (portal: acknowledge / raise discrepancy, own PO); Budget Owner / Finance / Management (view); Administrator.
- **Realizes:** narrative steps 9-10 (approve/reject PO), 13 (supplier acknowledges), 14-15 (advance payment trigger), 16-17 (pre-receipt amendment), 18 (hand off to fulfilment), and the framework call-off detail variant.
- **Purpose:** the single record of truth for one order through its lifecycle: header, lines, tax, the budget commitment, acknowledgement, amendments, and the PDF, with only the legal next actions shown.
- **Data shown:** all header + line fields; `PurchaseOrder.status` (Badge) and the `poLifecycle` position as a progress strip; the budget `Commitment` linked to the PO; `PurchaseOrder.acknowledged` + acknowledgedAt; `PurchaseOrder.editableUntilReceipt` (whether full edits are still allowed); `PurchaseOrder.frameworkAgreementId` (for a call-off PO, with the framework headroom); the issued PO PDF in `PurchaseOrder.attachments`; linked requisition / contract / advance payment; the append-only audit trail.
- **Primary actions (state-aware, per `poLifecycle`):** "Approve" / "Reject" (Approver, only when status = awaiting-approval; reject with a note returns to draft; auto-approve within the approver limit on `valueInBase`, axiom A7); "Acknowledge" (Supplier via portal, `po.acknowledge`; `poLifecycle` ISSUED -> ACKNOWLEDGED; effect: trigger advance payment per schedule on advance-bearing terms); "Raise discrepancy" (Supplier; returns to buyer); "Amend" (Buyer, `po.amend`; `poLifecycle` ACKNOWLEDGED -> ACKNOWLEDGED; guard: before first GRN, `editableUntilReceipt` true; audited; recompute commitment delta; revised PDF re-issued and re-acknowledged); "Cancel" (Buyer; DRAFT -> CANCELLED). Buttons render ONLY for legal transitions.
- **Secondary actions / navigation:** Tabs (Header / Lines and Tax / Budget and Commitment / Acknowledgement / Documents / Audit); download the PO PDF; link to GRN (08), invoice/match (09), payments (10), the framework (for call-offs).
- **States to design:** loading, populated, awaiting-approval (Approver sees Approve/Reject; creator does not), issued (awaiting supplier acknowledgement), acknowledged, amend-locked (after first GRN, full edits disabled, only qty-tolerance amend at receiving), terminal (closed / cancelled, read-only), permission-denied.
- **Shadcn components:** Card + Tabs, Badge (status, acknowledged, call-off), Stepper/progress strip (poLifecycle), Dialog (reject with note, amend, cancel), Button group, Toast, embedded PDF viewer/link.
- **Acceptance criteria (believable done):** (1) only legal `poLifecycle` transition buttons render for the current status (no Approve unless awaiting-approval; no Amend on a closed PO; no Acknowledge unless issued); (2) the PO creator cannot approve the PO (SoD, axiom A6) - the Approve action never renders for the creating buyer; (3) supplier acknowledgement (`poLifecycle` ISSUED -> ACKNOWLEDGED) triggers the advance payment only on advance-bearing terms (100% advance / part-advance against documents / part-advance against shipment); (4) amendment is allowed only before the first GRN (`editableUntilReceipt`); after first GRN full edits are locked and only the qty-tolerance amend at receiving applies; a value-increasing amend re-enters the budget and approval gates and re-issues/re-acknowledges the PDF; (5) `PurchaseOrder.number` is immutable once issued; (6) the audit trail is append-only and timestamped; for a call-off PO the framework ceiling draw-down is shown and SoD still applies to value-changing edits.
- **Tags:** [S2.1 / S3.1 / OE6 | ISO 8.4.3(d) / 7.5 | AB ack + amend-until-receipt + RA portal]

### Inline edge behaviours shown on these screens (not separate screens)
- **SoD self-approval block (axiom A6):** on S05.2 / S05.3 the approval engine excludes the PO creator from the eligible approver pool; the Approve/Issue action does not render for the creating buyer.
- **Budget hard-commit vs over-budget override (axiom A4 / OE11):** on S05.2 budget is hard-committed (encumbered) at issue, distinct from the soft check at requisition; an over-budget PO is blocked until the Budget Owner approves the override with a logged reason.
- **Incoterm validated vs transport mode:** on S05.2 FOB/CFR/CIF submitted for a non-sea mode is rejected; air/road/courier/multimodal use FCA/CPT/CIP/DAP; EXW is mode-agnostic.
- **Tax on lines (axiom A8 support):** on S05.2 `POLine.taxCodeId` makes tax a first-class line attribute (reverse-charge for imports) so the three-way match later reconciles amounts including tax.
- **PO type drives match path:** item -> three-way (PO vs GRN vs invoice); service/maintenance -> two-way + milestone acceptance, no GRN; freight ranks carriers and tracks AWB/BL.
- **Auto-approval within limit (axiom A7):** on S05.3 a PO whose `valueInBase` is within the approver limit auto-approves; a later pay-terms/schedule change can trigger a financial revert.
- **Currency degradation (e08):** on S05.2 if FX conversion fails, `PurchaseOrder.valueInBase` uses the original amount unconverted with a logged "rate unavailable" note; thresholds operate on the approximate base; never throws.
- **Call-off bypass:** a call-off PO (`frameworkAgreementId` set) draws down the framework ceiling without competitive approval, but SoD still applies to value-changing edits.
