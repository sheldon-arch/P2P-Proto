# Screens: 08 Delivery, Goods Receipt and Inspection (Build-real)

Flow 08 realized as screens. Source: `documentation/08-delivery-grn.md`, data dictionary `PartialDeliveryBlock.json` / `GoodsReceipt.json` / `Inspection.json` / `NCR.json` / `GrIrEntry.json`, `model/role-permission-matrix.md`, state machines `requisitionStage` / `ncrLifecycle`. Covers the Receiving landing, the inbound tracking and partial-delivery block, the Quality landing and inspection (COA hard block, reject to NCR), the GRN raise (quantity-tolerance PO amend, GR/IR accrual, spares no-GRN variant), and the derived delivery-status read.

### S08.1 Inbound / Expected Deliveries (archetype: list / per-role landing for Receiving)
- **Route:** `/inbound`
- **Roles:** Receiving/Warehouse (act: `inbound.track`, `delivery.record`); Buyer (view + record manual tracking, `inbound.track`); Tax/Compliance (view consignments awaiting customs, `inbound.track` conditional customs); requester/manager (same read view, view-only).
- **Realizes:** steps 1-10 (dispatch, transport-mode tracking, customs, ETA monitoring, overdue check, confirm arrival); the Receiving landing from storyboard section 6.
- **Purpose:** the receiving handler's home - every consignment expected against an acknowledged PO, where each one is in inbound logistics, what is overdue, and what is ready to receive. Answers "what is arriving and what do I act on" in the first 30 seconds.
- **Data shown:** per consignment from `PartialDeliveryBlock.json` - transportMode (Badge), carrier, trackingNumber_MAWB_HAWB, eta, actualDeliveryDate, boeNumber (import), deliveryStatus (derived Badge: delivered/partial/overdue/upcoming); the linked PO number (`GoodsReceipt.poReference`), supplier (`GoodsReceipt.supplierId`), requisition (`GoodsReceipt.requisitionReference`); the PO committed delivery date for the overdue comparison. Tracking is manual; no carrier-portal auto-fetch.
- **Primary actions:** "Record delivery / block" (-> S08.2; `delivery.record`); row click -> consignment detail; "Confirm arrival" (sets actualDeliveryDate, Receiving, `delivery.record`).
- **Secondary actions / navigation:** faceted filter by transportMode / deliveryStatus / overdue / import-vs-domestic; search by PO / tracking / supplier; link to the Quality inspection queue (S08.3); link to PO (flow 05). ETA-approaching alarm and overdue reminder surface as inline row Badges (fired by System; not user actions).
- **States to design:** empty ("No inbound consignments. Deliveries appear here once a PO is acknowledged."), loading (skeleton table), populated, error.
- **Shadcn components:** DataTable, Popover+Command (faceted filter), Input (search), Badge (transportMode, derived deliveryStatus with delivered green / partial amber / overdue red / upcoming grey), Button, Toast.
- **Acceptance criteria (believable done):** (1) deliveryStatus Badge is the derived value (delivered if aggregatedDelivered >= ordered with over-delivery counting; partial if 0 < delivered < ordered; overdue if delivered = 0 and needDate < start-of-today, null needDate never overdue; else upcoming), never a stored field; (2) the overdue Badge appears only when start-of-today is past the PO committed delivery date with nothing received, and a null committed date is never overdue; (3) the ETA-approaching Badge reflects the configurable lead (default 5 days before eta) and re-arms when eta is edited; (4) the row read view is identical for Receiving, Buyer, requester and manager (shared read, no role-specific data hiding).
- **Tags:** [S2.3/S2.4 | AB delivery-tracking + RA derived status]

### S08.2 Record Delivery / Partial-Delivery Block (archetype: form)
- **Route:** `/inbound/:poId/block/new`, `/inbound/block/:id/edit`
- **Roles:** Receiving/Warehouse (act: `delivery.record`); Buyer (manual inbound tracking fields, `inbound.track`); Tax/Compliance (customs block fields, `customs.manage` conditional import).
- **Realizes:** steps 2-7 (dispatch documents, transport-mode tracking block, customs/Bayan, ETA), 11-12 (full vs partial consignment, record one block per consignment).
- **Data shown / captured:** one `PartialDeliveryBlock.json` per arriving consignment - transportMode (Select {Air, Sea, Road, Courier}; drives the mode-specific field group and customs routing), carrier, courierCargo, trackingNumber_MAWB_HAWB, eta (editable, re-arms alarm), etaAlarmLeadDays, actualDeliveryDate (mandatory, gates stage advance), erpGrNumber, erpCcNumber, itemsDelivery (array of {itemId, quantity}; subset allowed; invalid entries skipped silently; null/empty becomes empty array). Mode-specific groups revealed by transportMode: Air (MAWB/HAWB, airline, flight, origin/destination IATA), Sea (BL, vessel, POL, POD; arrival confirmed only after the post-customs road-haulage leg), Road (LR number, transporter, roadOrigin {domestic, Saudi, UAE, other cross-border}), Courier (courier tracking, courier company; no customs). Customs block (Tax/Compliance, conditional on import = air/sea/cross-border road): boeNumber, clearanceBy, bayanDate, dutyAmount, homeMOOWR; ministryPermit (mandatory for regulated chemicals, alert on selection and ~1 week before expiry), healthCertificate (HARD mandatory for food RM), dgDeclaration (mandatory for dangerous goods).
- **Primary actions:** "Save block" (records the consignment block, Receiving, `delivery.record`); "Submit to customs" (routes the block to Tax/Compliance customs clearance for imports; condition transportMode in {Air, Sea} or cross-border Road). Recording one block does not COMPLETE the ticket; it advances PARTIAL_DELIVERY -> POST_DELIVERY only when the per-block mandatory fields are filled (`requisitionStage` moveStage; guard allMandatoryFieldsFilled per block).
- **Secondary actions / navigation:** add/remove itemsDelivery rows; edit eta (re-arms alarm, audited); attach shipment documents (AWB/BL/LR, Bayan, health cert, DG declaration per-shipment; COA/MSDS/certs permanent against the supplier master); domestic road skips the customs section.
- **States to design:** empty (new block), loading (edit fetch), validation-error (inline per field + remainingStageFields summary with the block index on stage-advance failure), saved toast, customs-routed toast.
- **Shadcn components:** Form (react-hook-form + Zod from `PartialDeliveryBlock.json`), Select (transportMode), conditional field groups (Card per mode), DatePicker (eta, actualDeliveryDate, bayanDate), Sheet (itemsDelivery line editor), file Input (documents), Button, Toast.
- **Acceptance criteria (believable done):** (1) transportMode reveals exactly its mode field group and routes air/sea/cross-border road to customs while courier and domestic road skip it (field-config by transportMode + roadOrigin); (2) itemsDelivery entries with a non-string id or non-finite quantity are skipped silently and null/empty becomes an empty array (no error), and delivered quantity aggregates per itemId across all blocks; (3) editing eta re-arms the ETA-approaching alarm and writes an audit entry; (4) healthCertificate is a hard mandatory for food RM imports and ministryPermit is mandatory for regulated chemicals (conditional customs gate); (5) the PARTIAL_DELIVERY -> POST_DELIVERY advance validates the per-block mandatory fields and returns remainingStageFields with the offending block index on failure, and advancing does not COMPLETE the ticket.
- **Tags:** [S2.3/S2.4 | ISO 8.4.3 | AB inbound-logistics + RA partial-delivery block]

### S08.3 QC Inspection Queue (archetype: queue / per-role landing for Quality)
- **Route:** `/quality/inspections`
- **Roles:** Quality (act: `qc.inspect`, `qc.approve`, `qc.reject`, `ncr.raise`); Buyer (view; chases COA, `inbound.track`).
- **Realizes:** steps 14-15 (store sends COA + sample, COA gate); the Quality landing from storyboard section 6 (lots awaiting inspection + open NCRs/CAPAs).
- **Purpose:** the inspector's home - materials lots awaiting verification, which are blocked for a missing COA, and the open NCRs they own. QC always inspects against the requisition; the PO is procurement-only and not shown here.
- **Data shown:** per lot from `Inspection.json` / `GoodsReceipt.json` - identifier, requisitionReference, supplierId, receivingStore, coaReceived (Badge), msdsReceived, sampleSent, inspectionLevel (Badge), inspectionDate, qcDecision (Badge once decided); a COA-blocked Badge when coaReceived = false; a tab of open NCRs from `NCR.json` (id, itemId, status, disposition).
- **Primary actions:** "Inspect" (-> S08.4; `qc.inspect`, enabled only when coaReceived = true and sampleSent = true); "Chase COA" (Buyer notification on the order thread, `inbound.track`; shown on COA-blocked rows). Spares/services lots never appear here (no inspection, no GRN).
- **Secondary actions / navigation:** filter by inspectionLevel / COA-blocked / decided-vs-pending / supplier; tab to open NCRs/CAPAs; search by lot / item / supplier.
- **States to design:** empty ("No lots awaiting inspection."), loading, populated, COA-blocked (rows visibly gated with the Inspect action disabled), error.
- **Shadcn components:** DataTable + Tabs (Awaiting inspection / Open NCRs), Badge (COA, inspectionLevel, qcDecision, NCR status), Button (disabled when COA-blocked), Toast.
- **Acceptance criteria (believable done):** (1) the Inspect action is disabled and a COA-blocked Badge shows whenever coaReceived = false, demonstrating the COA hard block (no inspection without the certificate, ISO 8.4.3(b)/8.6); (2) only materials-path lots appear (spares/services received by Engineering never enter the queue); (3) inspectionLevel reflects the risk-based level (skip-lot / sampling / 100% / source / CoC-only per ISO 8.4.2); (4) the queue shows lots against the requisition reference, never the PO (PO stays procurement-only).
- **Tags:** [S2.5 | ISO 8.4.2/8.6 | AB QC + factory-import COA gate]

### S08.4 Inspect Lot (archetype: form / modal)
- **Route:** `/quality/inspections/:id`
- **Roles:** Quality (act: `qc.inspect`, `qc.approve`, `qc.reject`, `ncr.raise`).
- **Realizes:** steps 17-19 (inspect and test sample, QC approved gate, reject raises NCR).
- **Purpose:** the verification screen where the inspector records the risk-based test result and the approve/reject decision that releases the lot to GRN or diverts it to an NCR; the single ISO 8.6 release-verification record.
- **Data shown / captured:** `Inspection.json` - coaReceived (read; gates), msdsReceived (conditional chemicals/DG), sampleSent, inspectionLevel (Select), qcDecision (Select {approve, reject}; gates), defectQuantity (>= 0, <= received), inspectionNotes (Textarea, max 2000), testAttachments (files), inspectionDate (default today, may be 10-15 days after receipt), inspectorId (auto, the authorizing person). On reject the screen opens the NCR sub-form from `NCR.json` - itemId, deliveryNoteNumber, deliveryDate, description (max 2000), percentNonConformance (0-100), image (>= 1 for defective/damaged/quality-fail), disposition (set later), decidingAuthority.
- **Primary actions:** "Approve lot" (qcDecision = approve; releases to the quantity-tolerance / GRN step S08.5; `qc.approve`; condition coaReceived = true); "Reject lot" (qcDecision = reject; raises an NCR `ncrLifecycle` RAISED via on=disposition entry, quarantines the goods, never produces a GRN; `qc.reject` + `ncr.raise`).
- **Secondary actions / navigation:** attach test reports; set inspectionLevel; link to the raised NCR (flow 11 / e04 disposition); back to the queue.
- **States to design:** loading, populated, COA-blocked (the whole approve path disabled until coaReceived = true), validation-error (NCR image required, percentNonConformance range), approved toast, NCR-raised toast.
- **Shadcn components:** Form, Select (qcDecision, inspectionLevel), Textarea, file Input, Dialog (reject confirmation opening the NCR sub-form), Badge, Button, Toast.
- **Acceptance criteria (believable done):** (1) Approve is blocked while coaReceived = false (COA hard block enforced at the decision, not just the queue); (2) a reject sets qcDecision = reject, opens the NCR with a mandatory image for defective/damaged/quality-fail, sets the NCR status RAISED and quarantines the lot, and never raises a GRN (axiom A11 entry to the CAPA loop); (3) an approve carries qcDecision = approve onto the GRN so only an approved lot becomes a receipt; (4) inspectorId is recorded as the authorizing person on the verification record (traceability-to-person, ISO 8.6); (5) inspectionNotes caps at 2000 and the result is retained as conformity evidence.
- **Tags:** [S2.5 | ISO 8.6 -> 8.7 | AB QC + e04 NCR]

### S08.5 Raise GRN (archetype: form)
- **Route:** `/inbound/:poId/grn/new`
- **Roles:** Receiving/Warehouse (act: `grn.raise`); Engineering (spares/services no-GRN closure variant, `grn.raise` conditional spares).
- **Realizes:** steps 20-23 (quantity variance to PO, amend PO quantity within tolerance, raise GRN, accrue GR/IR + relieve commitment), 24-26 (Engineering receives spares/services, close requisition no GRN).
- **Purpose:** the receipt event that turns an approved lot into a GRN, accrues the GR/IR liability, relieves the commitment to actual, and (for tolerance-eligible items) reconciles a quantity variance by amending the PO before the GRN so the three-way match clears. The materials path raises a GRN; the spares/services path closes the requisition with no GRN.
- **Data shown / captured:** `GoodsReceipt.json` - grnNumber (auto), poReference, requisitionReference, supplierId (auto from PO), receivedLines (array of {poLineId, acceptedQty >= 0, rejectedQty >= 0}; acceptedQty + rejectedQty <= delivered for the line), receivedQty (computed roll-up), receiptDate (default today), receivingStore, coaAttached (read; must be true), qcDecision (read; must be approve), erpGrNumber, raisedBy (auto), grnDate (auto). The quantity-variance panel compares actualReceivedQty against poLineQty and shows the per-line lineTolerancePercent. Spares/services variant captures the Engineering closure fields - receivedAgainstRequisition, conformsToOrder (gates), receivedAsPerOrder (true to close), completionDate.
- **Primary actions:** "Amend PO quantity" (Buyer, on a tolerance-eligible line with abs(delta)/poLineQty <= lineTolerancePercent and delta != 0; adjusts the PO line to actual before GRN; audited; `po.amend`); "Raise GRN" (Receiving; creates the GRN, accrues a `GrIrEntry` accruedAmount = GRN accepted qty * PO agreed price, relieves the commitment committed -> actual; `grn.raise`; condition coaAttached = true and qcDecision = approve on the materials path); "Close requisition received-as-per-order" (Engineering, spares/services; single boolean, no GRN, no GR/IR, forces a downstream two-way match; `grn.raise` conditional spares). A variance beyond lineTolerancePercent routes to NCR / buyer review (not silently accepted).
- **Secondary actions / navigation:** edit acceptedQty / rejectedQty per line; rejected quantity links to the NCR/return path; one PO can produce several GRNs (partial deliveries); link to the GR/IR entry.
- **States to design:** loading, populated, blocked (GRN disabled until coaAttached = true and qcDecision = approve), tolerance-amend confirmation, variance-beyond-tolerance warning routing to NCR, GRN-raised toast (with the accrued GR/IR amount), requisition-closed toast (spares).
- **Shadcn components:** Form, DataTable (received lines with the variance column), Dialog (amend-PO confirm, audited), Badge (tolerance status, GR/IR accrued), Button (gated), Toast.
- **Acceptance criteria (believable done):** (1) Raise GRN is blocked until coaAttached = true and qcDecision = approve (COA hard block + QC gate precede the receipt, axiom A11); (2) a tolerance-eligible line with a non-zero variance within lineTolerancePercent offers Amend PO (adjusts the PO line to actual before GRN, audited) while a no-tolerance line (lineTolerancePercent = 0) never amends and a variance beyond tolerance routes to NCR/review, never silent acceptance (the +/-5-10% labels/cartons rule); (3) raising the GRN accrues a GrIrEntry of GRN accepted quantity * PO agreed price and relieves the commitment to actual (GR/IR is the three-way backbone; the commitment moves committed -> actual); (4) GRN existence on the PO line forces a downstream THREE_WAY match, while the spares/services closure raises no GRN and forces a TWO_WAY match (axiom A8 set here); (5) the receivedLines accepted quantity is what the three-way match compares against (GRN accepted quantity, not ordered quantity).
- **Tags:** [S2.6 | ISO 8.6 | AB GRN + tolerance + build-new GR/IR]

### Inline edge behaviours shown on these screens (not separate screens)
- **Quantity tolerance amends PO before GRN (e03):** shown inline on S08.5 as the variance panel + the Amend PO action; labels/cartons/printed packaging amend within +/-5-10% (lineTolerancePercent), fixed-pack RM and contract glass bottles have lineTolerancePercent = 0 and never amend.
- **COA hard block (factory-import):** shown inline on S08.3 (Inspect disabled), S08.4 (Approve disabled) and S08.5 (Raise GRN disabled) whenever the COA is not on file; the buyer chases in a loop.
- **ETA-approaching alarm + overdue reminder:** shown inline on S08.1 as row Badges; fired by the System (timer), not user actions; an overdue delivery docks the supplier delivery score (feeds OTD / two-factor OTIF) and is non-blocking (the consignment can still be received late).
- **Derived delivery status (e):** shown inline on S08.1 as the deliveryStatus Badge, recomputed live, never stored; item and ticket precedence delivered > partial > overdue > upcoming.
- **Spares/services no-GRN variant:** shown inline on S08.5 as the Engineering closure card; closes the requisition with a single boolean, no GR/IR accrual, and forces a two-way match downstream.
